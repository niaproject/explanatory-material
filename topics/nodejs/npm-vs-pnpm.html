npm と pnpm の違い

同じ package.json を扱う 2 つのパッケージマネージャを、仕組み・速度・ディスク使用量・依存の厳格さで比較

結論（3 行まとめ）

  • npm = Node.js 公式・同梱。フラットな node_modules に依存を全部コピーするシンプル設計。
  • pnpm = Performant npm。グローバルストアに 1 回だけ 保存し ハードリンク + シンボリックリンク で各プロジェクトから参照する。
  • ディスク節約・インストール高速化・ファントム依存の防止なら pnpm。追加ツール不要の 素の Node.js 環境なら npm。

2 つの設計思想を 1 枚で

Standard
npm
Node.js に同梱。プロジェクトごとに依存ツリーを完全コピーする古典的アプローチ。
  • 追加インストール不要（Node.js 入れれば使える）
  • フラットな node_modules（依存の巻き上げ）
  • ロックファイル: package-lock.json
  • ワークスペース: npm v7+ で対応（機能は控えめ）
  • 間接依存も require できる（ファントム依存あり）
Performant
pnpm
CAS（Content-Addressable Store）に 1 度だけ保存し、各プロジェクトからリンクで共有。
  • 別途インストール（npm i -g pnpm または Corepack）
  • ネスト構造 + シンボリックリンクの node_modules
  • ロックファイル: pnpm-lock.yaml
  • ワークスペース・モノレポ機能が強力
  • 直接依存しか import できない（strict）

core の違い①: node_modules の構造

両者の最大の違いは node_modules の作り方です。同じ package.json（例: express だけに依存）でも中身の形が大きく変わります。

🔴 npm — フラット（hoisting）
node_modules/
├── express/           # 直接依存
├── accepts/           # 間接依存（巻き上げられた）
├── body-parser/       # 間接依存
├── cookie/            # 間接依存
├── debug/             # 間接依存
├── mime-types/        # 間接依存
└── …（数百個がフラットに並ぶ）
→ require("debug") が通ってしまう（実体は express の依存）。これがファントム依存。
🟡 pnpm — ネスト + symlink
node_modules/
├── express -> .pnpm/express@4.x/node_modules/express
└── .pnpm/
    ├── express@4.x/
    │   └── node_modules/
    │       ├── express/
    │       └── accepts -> ../../accepts@1.x/...
    ├── accepts@1.x/
    ├── body-parser@1.x/
    └── …
→ ルート直下にあるのは package.json に書かれた依存だけ。間接依存は .pnpm/ に隔離。
ポイント: pnpm の .pnpm/ 内の実体ファイルは、さらにグローバルストア （既定: ~/.local/share/pnpm/store や ~/Library/pnpm/store）への ハードリンクになっています。同じバージョンのパッケージはディスク上で 1 回しか保存されません。

core の違い②: ファントム依存（phantom dependencies）

「package.json に書いていないのに require/import できてしまう」状態をファントム依存と呼びます。本番でパッケージを外した瞬間に壊れる時限爆弾です。

npm の場合（通ってしまう）

package.json express
node_modules/ express debug accepts mime-types
require() require("express") ✓ require("debug") ✓ ← 危険

pnpm の場合（ブロックされる）

package.json express
node_modules/ express debug accepts
require() require("express") ✓ require("debug") ✗ Cannot find
現場で効く: 既存プロジェクトを npm → pnpm に切り替えた初日、 だいたい数個「見えなかったファントム依存」が Cannot find module で落ちます。 それは pnpm が壊したのではなく、前から隠れていた不整合が顕在化しただけです。

モノレポ（workspaces）の違い

両者とも workspaces をサポートしますが、成熟度と使い勝手に差があります。

npm workspaces（v7+）

// package.json（ルート）
{
  "workspaces": ["packages/*"]
}
  • 設定はシンプルだが、巻き上げ前提のためファントム依存が起きやすい。
  • フィルタ実行（特定パッケージだけ操作）は -w オプションで可能だが機能は限定的。

pnpm workspaces

# pnpm-workspace.yaml
packages:
  - "packages/*"
  - "apps/*"
  • pnpm -r（recursive）で全ワークスペース一括実行。
  • pnpm --filter <name> で依存関係まで考慮した絞り込みが強力。
  • 内部リンクは workspace:* プロトコルで明示でき、公開時に自動でバージョン解決される。
判断基準: パッケージ数が 10 を超えたり、Turborepo / Nx と組み合わせる本格モノレポなら pnpm ほぼ一択。単純な 2〜3 パッケージ構成なら npm workspaces でも回ります。

互換性・ハマりどころ

論点 npm pnpm
ファントム依存を拾うツール 動く（悪くも） 落ちる（良くも）
古い bundler（webpack v4 以前など） 無難 シンボリックリンク設定が必要な場合あり
Docker イメージのレイヤキャッシュ 標準的 --frozen-lockfile + store マウントで高速
Windows での symlink 不要 開発者モード or 管理者権限が望ましい
ロックファイルの互換 package-lock.json pnpm-lock.yaml（相互変換は非公式ツール）
注意: 一部の古いライブラリや設定（特に TypeScript の baseUrl/paths 解決、 node_modules を直接 grep する CLI など）は pnpm のリンク構造で挙動が変わることがあります。 問題が出たら .npmrc に node-linker=hoisted を書くと npm 風のフラット構造に寄せられます （ただし pnpm の旨味はほぼ消える）。

どちらを選ぶべきか

🔴 npm を選ぶべき
  • Node.js さえあれば動く環境を維持したい（CI の薄さ重視）
  • 小さな単一リポジトリ、依存も少ない
  • チームの大半が Node 初学者で学習コストを抑えたい
  • ツールチェーンが古く、シンボリックリンクで事故りそう
🟡 pnpm を選ぶべき
  • モノレポ・複数パッケージ構成（Turborepo / Nx と相性◎）
  • 同じ依存を何度もインストールしている（ディスク・CI 時間がもったいない）
  • ファントム依存を仕組みで防ぎたい（strict に寄せたい）
  • CI が遅くてビルド料金が痛い

pnpm の導入手順

方法 1: Corepack（推奨・Node.js 16.10+ に同梱）

corepack enable
corepack prepare pnpm@latest --activate

# プロジェクトで固定したいときは package.json に
// "packageManager": "pnpm@9.0.0"

方法 2: npm から入れる

npm install -g pnpm

方法 3: スタンドアロンインストーラ

# macOS / Linux
curl -fsSL https://get.pnpm.io/install.sh | sh -

# Windows (PowerShell)
iwr https://get.pnpm.io/install.ps1 -useb | iex
おすすめ: チーム開発なら Corepack + packageManager フィールド が最強。 誰の手元でも pnpm のバージョンが揃い、npm install してしまった人にも警告が出ます。

既存プロジェクトの移行チェックリスト

  • node_modules と package-lock.json を削除（rm -rf node_modules package-lock.json）
  • pnpm import で既存 package-lock.json から pnpm-lock.yaml を生成（精度高）
  • pnpm install を実行。Cannot find module が出たらファントム依存。pnpm add で正式に追加
  • CI スクリプトを置き換え（npm ci → pnpm install --frozen-lockfile）
  • .npmrc の確認（strict-peer-dependencies、auto-install-peers の挙動）
  • Docker イメージで Corepack or npm i -g pnpm を仕込む
  • package.json に packageManager を追記してチーム内で固定
  • README / CONTRIBUTING の npm install を全部置換

まとめ

  • npm は 同梱のシンプルさ、pnpm は リンク設計による高速化と strict さが売り。
  • 小規模・単一リポジトリ → npm、モノレポ・依存が多い・CI を速くしたい → pnpm。
  • 迷ったら corepack enable で 両方使える状態にしておき、新規プロジェクトから pnpm を試すのが低リスク。
  • コマンドの 9 割は対応表で置き換え可能なので、乗り換えコストはエコシステム体力より「ファントム依存の掃除」次第。
