git add を取り消す方法

ステージングした特定のファイルだけを取り消す完全ガイド

まず状況を整理しよう — ファイルが存在する 4 つのエリア

Git ではファイルが 4 つのエリアを行き来します。git add で誤ってステージしたファイルは「② Staging Area」に存在している状態です。取り消しとは、これを「① Working Directory」に戻す操作に他なりません。

git status が表示する 3 つの状態

上の 4 エリアのうち「① Working Directory」にあるファイルは、さらに 3 つの状態に分かれます。git status の出力はこの区分を示しています。

状態どのエリア?説明git status 表示
staged② Staging Area次のコミットに含まれるChanges to be committed
modified① Working Directory編集済みだがステージ前Changes not staged for commit
untracked① Working Directory一度も add されていない新規ファイルUntracked files
取り消し = エリアを 1 つ戻す操作: git add は ① → ② の移動。取り消しは ② → ① の逆移動です。ファイルの 中身 はどちらのエリアでも同じで、「Git から見たラベル」だけが変わります。
最初のアクション: 迷ったら必ず git status で現在の状態を確認しましょう。Git は親切にも、取り消し用のコマンドも提示してくれます。

どうやったら modified になるのか — ファイル状態の遷移

modified とは「一度 Git に追跡された (tracked) ファイルが、作業ツリー上で編集されて差分を持った状態」を指します。つまり untracked なファイルは直接 modified にはなりません。ここでは committed / staged / modified / untracked の関係を、循環図と 3 つの典型パターンで整理します。

重要な前提: Untracked → Modified の直接遷移は存在しません。新規ファイルは一度 git add → git commit を経て tracked になって初めて、次回の編集で modified を名乗れます。modified は「過去に Git が知っている状態」との差分のことだからです。
パターン A committed → modified（コミット済を編集）

もっとも基本のケース。以前コミットしたファイルを編集すると、作業ツリーと HEAD の間に差分が生まれて modified になります。

Before
● committed: app.js
→
After (ファイルを編集)
● modified: app.js
# コミット済ファイルに 1 行追記
$ echo "// new line" >> app.js
$ git status
Changes not staged for commit:
  modified:   app.js
パターン B staged → modified（add 後に再編集 / 同時保持）

git add した後でさらに同じファイルを編集した場合、staged な内容（add した瞬間のスナップショット）と modified な内容（その後の追加編集）の両方を同時に保持する状態になります。

Before (add 済み)
● staged: app.js
→
After (さらに編集)
● staged: app.js (add 時点の差分)
● modified: app.js (それ以降の差分)
# add した後でもう一度編集
$ git add app.js
$ echo "// more" >> app.js
$ git status
Changes to be committed:
  modified:   app.js
Changes not staged for commit:
  modified:   app.js   # ← 同じファイルが 2 箇所に！
ポイント: Git は「index（staged スナップショット）」と「作業ツリー」を別々に管理しているため、同一ファイルが両方に現れても矛盾しません。git diff が作業ツリーとの差分、git diff --staged が index との差分を見せてくれます。
パターン C untracked は modified にならない（反例）

新規作成したばかりのファイルは、Git から見れば「比較対象がない」ため差分が計算できません。したがって modified ではなく untracked として扱われます。

Before (新規作成)
● untracked: new-file.txt
→
After (まだ untracked のまま)
● untracked: new-file.txt
$ touch new-file.txt
$ git status
Untracked files:
  new-file.txt   # ← modified ではない
反例として覚える: 「ファイルを作った / 編集した」だけでは modified にはなりません。一度でも Git の追跡下に入ったか（= 過去に add → commit されたか）が分岐点です。新規ファイルは git add してはじめて Git の世界に入り、次回以降の編集で modified を名乗れるようになります。

確認コマンド 3 ステップ

git status — ファイルごとの状態ラベルを見る

どのファイルが staged / modified / untracked のどれなのかを一覧で確認します。まず最初にこれを打つのが鉄則です。

$ git status
Changes to be committed:      # ← staged
Changes not staged for commit:  # ← modified
Untracked files:               # ← untracked

git diff — 作業ツリーと index の差分（= 未 add の編集）

modified として表示されている部分、つまり「add 後に追加した編集」や「まだ add していない編集」の中身を確認します。

$ git diff           # modified な差分のみが見える

git diff --staged — index と HEAD の差分（= add 済みの内容）

staged として表示されている部分、つまり「次の git commit で履歴に残る内容」を確認します。パターン B の状況でも、この 2 つを併用すれば「どちらの差分か」をきれいに切り分けられます。

$ git diff --staged   # staged な差分のみが見える
# (別名: git diff --cached でも同じ)
まとめ: modified は「一度 Git に追跡されたファイルが、index または HEAD と異なる内容を作業ツリーに持つ状態」。untracked は追跡前なので modified にはなりません。add 後の再編集では staged と modified が同居し、git diff / git diff --staged でそれぞれの差分を別々に覗けます。

特定のファイルだけステージングを取り消す

推奨 ✅ git restore --staged <file> (Git 2.23+)

現在推奨されている方法。ファイルの中身はそのままに、ステージングだけを解除します。

# 1 ファイルだけ取り消す
$ git restore --staged index.html

# 複数ファイルをまとめて
$ git restore --staged index.html style.css

# ディレクトリ配下すべて
$ git restore --staged src/

# 全ステージングを取り消す
$ git restore --staged .
Before
● staged: index.html
● staged: style.css
→
After (index.html のみ取り消し)
● modified: index.html
● staged: style.css
ポイント: ファイルの 中身 は一切変更されません。ステージングだけが解除され、編集内容は作業ディレクトリに残ります。
従来 🔄 git reset HEAD <file> (古い Git でも使える)

Git 2.23 より前からある古くからの方法。動作は git restore --staged とほぼ同じです。

# 1 ファイルだけ取り消す
$ git reset HEAD index.html

# HEAD は省略可能
$ git reset index.html

# 全部まとめて取り消す
$ git reset
注意: git reset は使い方を誤ると履歴を壊す危険があります。ファイル指定なしで --hard を付けると作業内容が消えるので要注意。特定ファイルのステージ解除だけなら git restore --staged の方が安全です。

新規ファイル (Untracked) を add してしまった場合

初回の git add でステージに入れた新規ファイルを取り消したい場合も、手順は同じです。

# ステージから外す (ファイル自体はそのまま残る)
$ git restore --staged newfile.txt

# git status で確認すると「Untracked files」に戻っている
$ git status
Untracked files:
  newfile.txt
なぜ Untracked に戻るのか? 新規ファイルは元々 Git が知らないので、ステージ解除すると「Git が知らないファイル = Untracked」状態に戻ります。

手順まとめ (実践フロー)

現状を確認する

どのファイルがステージされているかを git status で確認します。

$ git status

取り消したいファイルを特定する

「Changes to be committed:」セクションに並んでいるファイルから、ステージから外したいものを決めます。

git restore --staged で取り消す

対象ファイルのパスを指定して実行します。

$ git restore --staged <file>

再度 git status で結果を確認

対象ファイルが「Changes not staged for commit」または「Untracked files」に移動していれば成功です。

$ git status

必要なファイルだけ改めて add する

コミットしたいファイルだけを選んで再度ステージに入れます。

$ git add <necessary-file>
$ git commit -m "意図したファイルだけをコミット"

コマンド比較早見表

コマンド 動作 作業ディレクトリ 推奨度
git restore --staged <file> 指定ファイルをステージから外す 変化なし ★★★ 最推奨
git reset HEAD <file> 同上 (古い書き方) 変化なし ★★ 互換用
git reset 全ファイルをステージから外す 変化なし ★★ 一括用
git restore <file> 作業ディレクトリの変更も破棄 変更が消える ★ 要注意
git reset --hard ステージも作業ディレクトリも全部破棄 すべて消える 最終手段
混同注意: git restore <file> (--staged なし) は作業ディレクトリのファイルを HEAD の状態まで戻してしまいます。編集内容が消えるので、ステージ取り消しのつもりで実行しないでください。

よくある質問

Q1 取り消すとファイルの中身は消えますか?

いいえ。git restore --staged や git reset HEAD <file> はステージング領域の状態だけを変更し、作業ディレクトリには手を付けません。編集内容は安全です。

Q2 まだ 1 度もコミットしていないリポジトリでも使えますか?

使えます。ただし最初のコミット前は HEAD が存在しないので、git reset HEAD <file> はエラーになる場合があります。その場合は git rm --cached <file> を使いましょう。

$ git rm --cached newfile.txt
Q3 ワイルドカードで複数ファイルを一度に指定できますか?

できます。シェルが展開するパターンも、Git がマッチさせるパスも両方使えます。

# *.log にマッチするステージ済みファイルを取り消す
$ git restore --staged "*.log"
