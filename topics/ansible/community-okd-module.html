結論（TL;DR）

3 行でまとめると

community.okd は kubernetes.core を継承して OpenShift 固有の API（route.openshift.io, build.openshift.io, project.openshift.io, user.openshift.io, security.openshift.io など）を扱えるようにした公式コミュニティコレクションです。

「oc 専用モジュールが用意されているのは、CRUD で表現できないコマンドだけ」です。oc login（OAuth トークン取得）、oc adm prune（複数リソースの一括削除）、oc process（Template 展開）、oc adm migrate-template-instances といったサブリソース系や複合操作のみ専用モジュールが存在し、それ以外は kubernetes.core.k8s / k8s_info / k8s_drain をそのまま使います。

つまり oc コマンドを Playbook 化するときの判断は 「単純な GET / POST / PATCH / DELETE → k8s 系」「OAuth・サブリソース・複合操作 → openshift_* / okd 系」の 2 択で迷う必要がありません。

community.okd とは — 設計思想と kubernetes.core との関係

community.okd は Ansible Galaxy 上の OpenShift / OKD 用コレクションで、 内部的には kubernetes.core を 依存関係として取り込み、 OpenShift 固有の処理だけを追加実装しています。よって community.okd.k8s は 実体としては kubernetes.core.k8s の薄いラッパーで、引数も挙動もほぼ同じです。

提供元

Red Hat 主導の openshift/community.okd リポジトリ。OpenShift 4.x / OKD 4.x をターゲット。

依存

制御ノードに kubernetes.core と Python の kubernetes / openshift / jsonpatch ライブラリ。

通信方式

oc バイナリは使わず、API Server の REST を直接呼ぶ。aggregated API 経由で K8s リソースと OpenShift リソースを区別なく扱える。

使い分けの鉄則

CRUD で表現できる操作 → kubernetes.core。OAuth / 複合 / サブリソース → community.okd。

提供される OpenShift 専用モジュール（抜粋）

認証 / 認可

  • community.okd.openshift_auth — oc login 相当。OAuth Implicit Grant でトークンを取得し、後続タスクで api_key として再利用。

ネットワーク

  • community.okd.openshift_route — oc expose 相当。Service を選んで Route を作成し、TLS Termination や Wildcard Policy も引数で指定。

テンプレート

  • community.okd.openshift_process — oc process 相当。Template の Parameter を埋め込んで展開、そのまま適用も可能。

イメージ

  • community.okd.openshift_import_image — oc import-image 相当。外部レジストリから ImageStream へ即時インポート。

oc adm 系（管理コマンド）

  • community.okd.openshift_adm_prune_deployments — 古い ReplicationController / ReplicaSet を一括削除。
  • community.okd.openshift_adm_prune_builds — 古い Build を一括削除。
  • community.okd.openshift_adm_prune_images — 未参照の Image / ImageStreamTag を削除。
  • community.okd.openshift_adm_prune_auth — 削除されたユーザー / グループへの参照を RoleBinding / SCC から除去。
  • community.okd.openshift_adm_migrate_template_instances — 旧 Template API の参照を新 GVK に更新。
  • community.okd.openshift_adm_groups_sync — LDAP との Group 同期。

インベントリ / ルックアップ

  • community.okd.openshift（インベントリプラグイン）— Pod を Ansible のインベントリとして自動列挙。

インストールと前提

# コレクションのインストール
ansible-galaxy collection install community.okd

# Python 依存（制御ノードで実行する Python に対して）
pip install kubernetes openshift PyYAML jsonpatch

# 動作確認
ansible-doc community.okd.openshift_auth | head -20
推奨セットアップ

Execution Environment（ansible-builder）を使うなら、execution-environment.yml の collections に kubernetes.core と community.okd を、python 依存に kubernetes / openshift / jsonpatch を追加するだけで再現可能な実行環境になります。

oc login ↔ openshift_auth

oc login は OpenShift の Internal OAuth Server に対する Implicit Grant で Bearer Token を取得し、それを kubeconfig に保存します。community.okd.openshift_auth は 同じフローを Python で再現するモジュールで、取得したトークンを以降のタスクに api_key として渡せます。

1ユーザー名 / パスワードでログイン
oc CLI
oc login https://api.cluster.example.com:6443 \
   --username=alice \
   --password='********' \
   --certificate-authority=/etc/ssl/ca.crt
成功すると ~/.kube/config に context と token が書き込まれ、以降の oc 実行で再利用される。
Ansible Playbook
- name: Log in to OpenShift
  community.okd.openshift_auth:
    host: https://api.cluster.example.com:6443
    username: alice
    password: "{{ ocp_password }}"
    ca_cert: /etc/ssl/ca.crt
  register: ocp
  no_log: true
レスポンスは ocp.openshift_auth.api_key に格納される。以降のタスクで api_key: "{{ ocp.openshift_auth.api_key }}" を渡すだけ。
2既存トークンでログイン
oc CLI
oc login https://api.cluster.example.com:6443 \
   --token=sha256~xxxxxxxxxxxxxxxxxx
Service Account のトークンや、すでに取得済みのトークンを直接指定するパターン。
Ansible Playbook
# openshift_auth を呼ばず、直接 api_key を渡せばよい
- name: Use existing token
  kubernetes.core.k8s_info:
    host: https://api.cluster.example.com:6443
    api_key: "{{ ocp_token }}"
    ca_cert: /etc/ssl/ca.crt
    kind: Project
K8S_AUTH_API_KEY 環境変数で渡すこともできる。CI/CD では Vault や Secret に保存して注入。
3ログアウト（トークン失効）
oc CLI
oc logout
取得した OAuth Access Token を /oauth/tokens エンドポイントで削除し、kubeconfig からも消す。
Ansible Playbook
- name: Revoke OpenShift token
  community.okd.openshift_auth:
    state: absent
    host: "{{ ocp.openshift_auth.host }}"
    api_key: "{{ ocp.openshift_auth.api_key }}"
    ca_cert: /etc/ssl/ca.crt
本番運用では Playbook 末尾の always 句に置くと、失敗時も確実に失効できる。

oc get pod ↔ k8s_info（Pod）

リソースの参照は OpenShift 固有のものを含めて すべて kubernetes.core.k8s_info で取れます。 oc get のフィルタ（label / field / namespace）はそのまま引数に対応します。

1特定 namespace の Pod 一覧
oc CLI
oc get pod -n myproject
Ansible Playbook
- name: List pods in myproject
  kubernetes.core.k8s_info:
    api_version: v1
    kind: Pod
    namespace: myproject
    api_key: "{{ ocp.openshift_auth.api_key }}"
  register: pods
2ラベルセレクタで絞り込み
oc CLI
oc get pod -n myproject \
   -l app=web,tier=frontend
Ansible Playbook
- name: List frontend pods
  kubernetes.core.k8s_info:
    kind: Pod
    namespace: myproject
    label_selectors:
      - app=web
      - tier=frontend
  register: frontend_pods
3フィールドセレクタで絞り込み（Running の Pod のみ）
oc CLI
oc get pod -n myproject \
   --field-selector=status.phase=Running
Ansible Playbook
- name: List running pods
  kubernetes.core.k8s_info:
    kind: Pod
    namespace: myproject
    field_selectors:
      - status.phase=Running
  register: running_pods
4全 namespace の Pod 一覧
oc CLI
oc get pod --all-namespaces
Ansible Playbook
- name: List pods cluster-wide
  kubernetes.core.k8s_info:
    kind: Pod
    # namespace を省略 = 全 namespace
  register: all_pods

- debug:
    msg: "Total pods: {{ all_pods.resources | length }}"
5Pod ログを取得
oc CLI
oc logs web-7d8f-xxxxx \
   -n myproject \
   --tail=100
Ansible Playbook
- name: Fetch pod logs
  kubernetes.core.k8s_log:
    name: web-7d8f-xxxxx
    namespace: myproject
    tail_lines: 100
  register: logs

oc get node ↔ k8s_info（Node）

Node はクラスタスコープのリソースなので、namespace 引数は不要です。 oc adm top node 相当のメトリクスは metrics.k8s.io API を見れば取れます（後述）。

1全ノード一覧
oc CLI
oc get node
Ansible Playbook
- name: List all nodes
  kubernetes.core.k8s_info:
    api_version: v1
    kind: Node
  register: nodes

- debug:
    msg: "{{ item.metadata.name }} : {{ item.status.conditions
            | selectattr('type', 'eq', 'Ready')
            | map(attribute='status') | first }}"
  loop: "{{ nodes.resources }}"
2master/worker ロール別に取得
oc CLI
oc get node \
   -l node-role.kubernetes.io/worker=
Ansible Playbook
- name: List worker nodes
  kubernetes.core.k8s_info:
    kind: Node
    label_selectors:
      - node-role.kubernetes.io/worker
  register: workers
値が空のラベルは「キーの存在」だけを判定する。Ansible 側も値なしで OK。
3NotReady な Node を抽出
oc CLI
oc get node \
   -o jsonpath='{range .items[?(@.status.conditions[?(@.type=="Ready")].status!="True")]}{.metadata.name}{"\n"}{end}'
Ansible Playbook
- name: Find NotReady nodes
  kubernetes.core.k8s_info:
    kind: Node
  register: all_nodes

- set_fact:
    notready: "{{ all_nodes.resources
              | rejectattr('status.conditions', 'search', 'Ready.*True')
              | map(attribute='metadata.name') | list }}"

- fail:
    msg: "NotReady nodes: {{ notready }}"
  when: notready | length > 0
jsonpath を書くより Jinja2 フィルタで書いたほうがロール・通知に組み込みやすい。

oc get project / route / dc / bc など

OpenShift 固有リソースも api_version を指定すれば k8s_info で取得できます。

oc コマンドapi_versionkind用途
oc get projectproject.openshift.io/v1Projectユーザー視点のプロジェクト一覧。
oc get routeroute.openshift.io/v1Route外部公開 URL の一覧。
oc get dcapps.openshift.io/v1DeploymentConfigレガシー。新規は Deployment 推奨。
oc get bcbuild.openshift.io/v1BuildConfigS2I / Docker ビルド定義。
oc get isimage.openshift.io/v1ImageStreamイメージタグの参照管理。
oc get useruser.openshift.io/v1UserOpenShift User オブジェクト。
oc get groupuser.openshift.io/v1GroupGroup オブジェクト。
oc get sccsecurity.openshift.io/v1SecurityContextConstraintsSCC 一覧。
oc get clusterversionconfig.openshift.io/v1ClusterVersionクラスタのバージョン情報。
例Route 一覧と外部 URL の抽出
oc CLI
oc get route -n myproject \
   -o jsonpath='{range .items[*]}{.spec.host}{"\n"}{end}'
Ansible Playbook
- name: List external URLs
  kubernetes.core.k8s_info:
    api_version: route.openshift.io/v1
    kind: Route
    namespace: myproject
  register: routes

- debug:
    msg: "{{ routes.resources | map(attribute='spec.host') | list }}"

oc adm cordon / drain / uncordon ↔ k8s_drain

ノードの保守オペレーションは kubernetes.core.k8s_drain 一本で cordon / drain / uncordon を切り替えます。 引数 state がそのまま 3 つの操作に対応します。

1cordon（新規スケジュールを停止）
oc CLI
oc adm cordon worker-1.example.com
Pod 退避はせず、新しい Pod の割り当てだけを止める。
Ansible Playbook
- name: Cordon worker-1
  kubernetes.core.k8s_drain:
    name: worker-1.example.com
    state: cordon
2drain（Pod を退避してメンテへ）
oc CLI
oc adm drain worker-1.example.com \
   --ignore-daemonsets \
   --delete-emptydir-data \
   --grace-period=600 \
   --force
Ansible Playbook
- name: Drain worker-1
  kubernetes.core.k8s_drain:
    name: worker-1.example.com
    state: drain
    delete_options:
      ignore_daemonsets: true
      delete_emptydir_data: true
      terminate_grace_period: 600
      force: true
PDB を尊重して順次 evict。タイムアウトは wait_timeout で上書き可能。
3uncordon（メンテ後にスケジュール再開）
oc CLI
oc adm uncordon worker-1.example.com
Ansible Playbook
- name: Uncordon worker-1
  kubernetes.core.k8s_drain:
    name: worker-1.example.com
    state: uncordon
運用 Tips

ローリングメンテで Worker を順に再起動するなら、kubernetes.core.k8s_info で node-role.kubernetes.io/worker ラベルを持つ Node を列挙し、loop で 1 台ずつ cordon → drain → 再起動（OS 側）→ uncordon を回すと安全です。

oc adm policy add-role / add-scc ↔ k8s（RoleBinding / SCC patch）

oc adm policy 系は専用モジュールがありません。「Role 付与は RoleBinding を作る」「SCC 付与は SCC を patch する」 と理解すれば kubernetes.core.k8s / k8s_json_patch で完結します。

1ユーザーに ClusterRole を namespace スコープで付与
oc CLI
oc adm policy add-role-to-user \
   admin alice \
   -n myproject
Ansible Playbook
- name: Grant admin role to alice on myproject
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: rbac.authorization.k8s.io/v1
      kind: RoleBinding
      metadata:
        name: admin-alice
        namespace: myproject
      subjects:
        - kind: User
          name: alice
          apiGroup: rbac.authorization.k8s.io
      roleRef:
        kind: ClusterRole
        name: admin
        apiGroup: rbac.authorization.k8s.io
命名は冪等のため固定化（{role}-{user} が安全）。
2ServiceAccount を SCC に追加
oc CLI
oc adm policy add-scc-to-user \
   privileged \
   -z default \
   -n myproject
-z は ServiceAccount 指定の短縮形。実体は SCC の users 配列に system:serviceaccount:myproject:default を追記。
Ansible Playbook
- name: Add SA to privileged SCC
  kubernetes.core.k8s_json_patch:
    api_version: security.openshift.io/v1
    kind: SecurityContextConstraints
    name: privileged
    patch:
      - op: add
        path: /users/-
        value: system:serviceaccount:myproject:default
RFC 6902 JSON Patch で配列末尾追加。重複追加を避けたいなら事前に k8s_info で確認 → when で分岐。
3クラスタ全体に ClusterRoleBinding を付与
oc CLI
oc adm policy add-cluster-role-to-user \
   cluster-admin alice
Ansible Playbook
- name: Promote alice to cluster-admin
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: rbac.authorization.k8s.io/v1
      kind: ClusterRoleBinding
      metadata:
        name: cluster-admin-alice
      subjects:
        - kind: User
          name: alice
          apiGroup: rbac.authorization.k8s.io
      roleRef:
        kind: ClusterRole
        name: cluster-admin
        apiGroup: rbac.authorization.k8s.io
注意

SCC を k8s モジュールで state: present のまま渡すと users 配列が丸ごと上書きされる事故が起きます。「既存ユーザーを残したまま追加」したい場合は必ず k8s_json_patch（差分追加）か Server-Side Apply の所有権分離（apply: true）を使ってください。

oc adm groups ↔ k8s（Group リソース）

OpenShift の Group は user.openshift.io/v1 の CRUD 可能なリソースなので、普通に k8s モジュールで操作できます。

1新規 Group の作成
oc CLI
oc adm groups new platform-admins \
   alice bob
Ansible Playbook
- name: Create platform-admins group
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: user.openshift.io/v1
      kind: Group
      metadata:
        name: platform-admins
      users:
        - alice
        - bob
2Group にユーザー追加
oc CLI
oc adm groups add-users \
   platform-admins carol
Ansible Playbook
- name: Add carol to platform-admins
  kubernetes.core.k8s_json_patch:
    api_version: user.openshift.io/v1
    kind: Group
    name: platform-admins
    patch:
      - op: add
        path: /users/-
        value: carol
3LDAP からの自動同期
oc CLI
oc adm groups sync \
   --sync-config=sync.yaml \
   --confirm
Ansible Playbook
- name: Sync groups from LDAP
  community.okd.openshift_adm_groups_sync:
    config_file: /etc/openshift/sync.yaml
    state: present     # 実反映（CLI の --confirm 相当）
state: info ならドライラン相当（CLI の --confirm なし）。

oc adm prune ↔ openshift_adm_prune_*

prune 系は 「複数リソースを条件付きで一括削除する」複合操作のため、専用モジュールが用意されています。 内部で k8s_info 相当の列挙 → 条件フィルタ → 削除をまとめて実行します。

1古い Deployment / RC のクリーンアップ
oc CLI
oc adm prune deployments \
   --keep-younger-than=60m \
   --keep-complete=5 \
   --keep-failed=1 \
   -n myproject \
   --confirm
Ansible Playbook
- name: Prune old deployments
  community.okd.openshift_adm_prune_deployments:
    namespace: myproject
    keep_younger_than: 60      # 分単位
    orphans: true             # 親のいない RC も対象
モジュールは常に --confirm 相当で実行。プレビューしたい場合は check_mode: true。
2古い Build のクリーンアップ
oc CLI
oc adm prune builds \
   --keep-younger-than=120m \
   --keep-complete=3 \
   --orphans \
   --confirm
Ansible Playbook
- name: Prune old builds cluster-wide
  community.okd.openshift_adm_prune_builds:
    keep_younger_than: 120
    orphans: true
3未参照イメージのクリーンアップ
oc CLI
oc adm prune images \
   --keep-younger-than=60m \
   --keep-tag-revisions=3 \
   --confirm \
   --registry-url=image-registry.openshift-image-registry.svc:5000
Ansible Playbook
- name: Prune unused images
  community.okd.openshift_adm_prune_images:
    keep_younger_than: 60
    registry_url: image-registry.openshift-image-registry.svc:5000
    ignore_invalid_refs: true
4削除されたユーザー / グループへの参照を除去
oc CLI
oc adm prune auth users deleted-user
Ansible Playbook
- name: Strip deleted user from RoleBindings / SCC
  community.okd.openshift_adm_prune_auth:
    resource: users
    name: deleted-user
RoleBinding / ClusterRoleBinding / SCC の subjects / users から自動除去。LDAP 同期から外れたユーザーの後始末に便利。

oc adm top ↔ metrics.k8s.io 経由

oc adm top node / oc adm top pod は Metrics Server の API（metrics.k8s.io/v1beta1）を叩いているだけです。 モジュールは存在しませんが、k8s_info で同じ API を叩けば取得できます。

1ノードの CPU / メモリ消費
oc CLI
oc adm top node
Ansible Playbook
- name: Fetch node metrics
  kubernetes.core.k8s_info:
    api_version: metrics.k8s.io/v1beta1
    kind: NodeMetrics
  register: nm

- debug:
    msg: "{{ item.metadata.name }}: cpu={{ item.usage.cpu }}, mem={{ item.usage.memory }}"
  loop: "{{ nm.resources }}"
2Pod の CPU / メモリ消費
oc CLI
oc adm top pod -n myproject
Ansible Playbook
- name: Fetch pod metrics
  kubernetes.core.k8s_info:
    api_version: metrics.k8s.io/v1beta1
    kind: PodMetrics
    namespace: myproject
  register: pm
クラスタに Metrics Server が無いと取得できない点は oc adm top と同じ。

oc adm must-gather / node-logs（モジュール非対応）

これらはローカルファイルシステムへの大規模ファイル収集がメインで、API として表現しにくいため、 community.okd も kubernetes.core も専用モジュールを提供していません。 現実解は「oc バイナリを command モジュール経由で呼ぶ」ことです。

1must-gather でクラスタ診断を取得
oc CLI
oc adm must-gather \
   --dest-dir=/tmp/mg
Ansible Playbook
- name: Run must-gather
  ansible.builtin.command:
    cmd: oc adm must-gather --dest-dir={{ mg_dir }}
    creates: "{{ mg_dir }}/version"
  environment:
    KUBECONFIG: "{{ kubeconfig_path }}"
  register: mg
前提として制御ノードに oc バイナリが必要。creates で簡易な冪等性を確保。
2node-logs（journald / kubelet ログ）
oc CLI
oc adm node-logs worker-1 \
   --unit=kubelet \
   --since="2 hours ago"
Ansible Playbook
- name: Fetch kubelet logs
  ansible.builtin.command:
    cmd: >-
      oc adm node-logs {{ node }}
      --unit=kubelet
      --since="2 hours ago"
  register: kubelet_log
  changed_when: false
純粋な「読み取り」なので changed_when: false を付けて意図を明示。
割り切り

oc adm must-gather は内部で Job を立ち上げ → 完了待ち → kubectl cp でファイル回収というマルチステップ複合動作です。これを純 API で書くのは現実的でないため、command モジュールで oc 自体を呼ぶのが定石です。逆に言えば、ここだけは「制御ノードに oc をインストールしておく」運用判断が必要です。

oc new-project / oc expose / oc process

高レベルコマンドのうち、頻出するものは community.okd に専用モジュールがあります。

1oc new-project（ProjectRequest の特殊扱い）
oc CLI
oc new-project myproject \
   --display-name="My Project" \
   --description="team A workload"
Ansible Playbook
- name: Create project
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: project.openshift.io/v1
      kind: ProjectRequest
      metadata:
        name: myproject
      displayName: "My Project"
      description: "team A workload"
Project ではなく ProjectRequest を POST する点に注意。OpenShift が自動で関連 RoleBinding / SCC を生成。
2oc expose svc（Route の作成）
oc CLI
oc expose svc/web \
   --hostname=web.apps.example.com \
   -n myproject
Ansible Playbook
- name: Expose web service
  community.okd.openshift_route:
    namespace: myproject
    service: web
    hostname: web.apps.example.com
    tls:
      termination: edge
      insecure_policy: Redirect
TLS Termination や Wildcard Policy も引数で指定できる。
3oc process（Template 展開）
oc CLI
oc process -f template.yaml \
   -p NAMESPACE=myproject \
   -p IMAGE=nginx:1.27 \
   | oc apply -f -
Ansible Playbook
- name: Process and apply template
  community.okd.openshift_process:
    src: files/template.yaml
    namespace: myproject
    parameters:
      NAMESPACE: myproject
      IMAGE: nginx:1.27
    state: present      # 展開後そのまま適用
state: rendered にすると展開結果だけ返して適用しない（ドライラン代わり）。

全部入りサンプル Playbook

ログイン → 状態確認 → ワーカーローリングメンテ → クリーンアップ → ログアウトの典型シナリオ。

---
- name: OpenShift maintenance workflow
  hosts: localhost
  connection: local
  gather_facts: false
  vars:
    ocp_host: https://api.cluster.example.com:6443
    ocp_ca:   /etc/ssl/ca.crt
    ocp_user: alice

  tasks:
    # --- 1. ログイン ---
    - name: Login (oc login 相当)
      community.okd.openshift_auth:
        host: "{{ ocp_host }}"
        username: "{{ ocp_user }}"
        password: "{{ ocp_password }}"
        ca_cert: "{{ ocp_ca }}"
      register: ocp
      no_log: true

    # --- 2. 健全性確認 (oc get node 相当) ---
    - name: List worker nodes
      kubernetes.core.k8s_info:
        host: "{{ ocp_host }}"
        api_key: "{{ ocp.openshift_auth.api_key }}"
        ca_cert: "{{ ocp_ca }}"
        kind: Node
        label_selectors:
          - node-role.kubernetes.io/worker
      register: workers

    # --- 3. ローリングメンテ (cordon → drain → uncordon) ---
    - name: Rolling maintenance per worker
      include_tasks: tasks/maintain_node.yml
      loop: "{{ workers.resources | map(attribute='metadata.name') | list }}"
      loop_control:
        loop_var: node_name

    # --- 4. クリーンアップ (oc adm prune 相当) ---
    - name: Prune old deployments cluster-wide
      community.okd.openshift_adm_prune_deployments:
        host: "{{ ocp_host }}"
        api_key: "{{ ocp.openshift_auth.api_key }}"
        ca_cert: "{{ ocp_ca }}"
        keep_younger_than: 120

    - name: Prune old builds
      community.okd.openshift_adm_prune_builds:
        host: "{{ ocp_host }}"
        api_key: "{{ ocp.openshift_auth.api_key }}"
        ca_cert: "{{ ocp_ca }}"
        keep_younger_than: 240

  # --- 5. 最後に必ずログアウト ---
  post_tasks:
    - name: Logout (oc logout 相当)
      community.okd.openshift_auth:
        state: absent
        host: "{{ ocp_host }}"
        api_key: "{{ ocp.openshift_auth.api_key }}"
        ca_cert: "{{ ocp_ca }}"
      when: ocp.openshift_auth.api_key is defined

tasks/maintain_node.yml

- name: Cordon {{ node_name }}
  kubernetes.core.k8s_drain:
    host: "{{ ocp_host }}"
    api_key: "{{ ocp.openshift_auth.api_key }}"
    ca_cert: "{{ ocp_ca }}"
    name: "{{ node_name }}"
    state: cordon

- name: Drain {{ node_name }}
  kubernetes.core.k8s_drain:
    host: "{{ ocp_host }}"
    api_key: "{{ ocp.openshift_auth.api_key }}"
    ca_cert: "{{ ocp_ca }}"
    name: "{{ node_name }}"
    state: drain
    delete_options:
      ignore_daemonsets: true
      delete_emptydir_data: true
      terminate_grace_period: 600

- name: Reboot {{ node_name }} (外部 IaaS 経由など、別途実装)
  ansible.builtin.debug:
    msg: "<ここで OS リブートや AMI 更新を実行>"

- name: Wait until node Ready again
  kubernetes.core.k8s_info:
    host: "{{ ocp_host }}"
    api_key: "{{ ocp.openshift_auth.api_key }}"
    ca_cert: "{{ ocp_ca }}"
    kind: Node
    name: "{{ node_name }}"
  register: n
  until: n.resources[0].status.conditions
        | selectattr('type', 'eq', 'Ready')
        | map(attribute='status') | first == "True"
  retries: 60
  delay: 10

- name: Uncordon {{ node_name }}
  kubernetes.core.k8s_drain:
    host: "{{ ocp_host }}"
    api_key: "{{ ocp.openshift_auth.api_key }}"
    ca_cert: "{{ ocp_ca }}"
    name: "{{ node_name }}"
    state: uncordon
設計のポイント

接続情報（host / api_key / ca_cert）はタスクごとに繰り返し書く代わりに、module_defaults でまとめて指定するとさらに DRY になります。module_defaults: { group/kubernetes.core.k8s: { host: ..., api_key: ..., ca_cert: ... } } のように書けば、配下のタスクから引数を省略できます。
