結論（TL;DR）

3 行でまとめると

Ansible の k8s モジュールは kubernetes.core コレクションに同梱された Python モジュールで、内部的には公式 kubernetes Python クライアント（OpenAPI 由来の dynamic client）を介して kube-apiserver の REST APIを直接叩きます。kubectl バイナリは 使いません。

Ansible の哲学である宣言的・冪等を満たすため、state: present ではまず GET で現在状態を取得し、diff を計算してから PATCH（または Server-Side Apply の apply）を発行します。差分がなければ changed=false を返します。

接続情報（kubeconfig / トークン / context / namespace）はモジュール引数または K8S_AUTH_* 環境変数で指定し、対象クラスタの kube-apiserver と TCP/TLS で会話できる Ansible 制御ノード上でモジュールが実行されます（リモートホスト経由でなくても OK）。

k8s モジュールとは何か

k8s モジュールは kubernetes.core という Ansible コレクションに含まれる Kubernetes リソース操作モジュール群の中心です。 Ansible 2.10 でコレクション分離が行われた際に旧 community.kubernetes から名前を変えて Red Hat 公式として再編成された経緯があります。 Ansible Galaxy または ansible-builder 経由でインストールします。 

# コレクションのインストール
ansible-galaxy collection install kubernetes.core

# 依存 Python クライアント（制御ノードに必須）
pip install kubernetes PyYAML jsonpatch

提供元

Red Hat / Ansible 公式の kubernetes.core コレクション。

通信方式

kube-apiserver の REST API を直接呼び出し（HTTPS / TLS）。

必須依存

制御ノードの Python に kubernetes（公式クライアント）。

扱える対象

API Server が認識する全リソース（CRD を含む）。

アーキテクチャ — Ansible → Python → kube-apiserver

一般的な Ansible モジュール（shell, file, apt など）は SSH 経由でリモートノードに転送されて実行されますが、 k8s モジュールは少し特殊で、「クラスタとネットワークで会話できるノード」に対して実行します。 多くの場合これは delegate_to: localhost された制御ノード自身です。

k8s モジュール実行時のレイヤ構造
Playbook / Role YAML タスク定義 — definition / src / state を渡す
▼ ansible-core が Python モジュールを転送・実行
kubernetes.core.k8s モジュール（Python） 引数バリデーション / 認証情報の解決 / リソース差分計算
▼ import kubernetes
kubernetes Python クライアント（dynamic client） OpenAPI Discovery / urllib3 で HTTPS リクエスト構築
▼ HTTPS (TLS) / Bearer Token または mTLS
kube-apiserver 認証 → 認可（RBAC）→ Admission → etcd 書き込み
▼ controller-manager / scheduler / kubelet が後続処理
etcd / 各ノードのコンテナランタイム 実際のリソース反映（Pod 起動 / Service ルーティング等）
ポイント

Ansible が直接話しかけるのは kube-apiserver だけです。Pod の起動・スケジューリングといった実体反映は Kubernetes 側のコントローラー責務で、Ansible は「宣言した状態を API に届ける」役割に徹します。

1 タスクが実行されるまでの流れ

state: present で Deployment を作成・更新する場合の内部処理：

Playbook が引数を解決 definition / src / template から最終的なマニフェスト dict を組み立て、認証情報（kubeconfig / host / api_key）を確定。
kubernetes クライアントを初期化 config.load_kube_config() または config.load_incluster_config() 相当を呼び、DynamicClient を生成。API Discovery で apiVersion + kind から正しい REST パスを解決。
GET で現状を取得 同 namespace / 同名のリソースが既に存在するかを GET /apis/<group>/<ver>/namespaces/<ns>/<plural>/<name> で確認。
差分を計算 取得した現状と望ましい状態を jsonpatch または Server-Side Apply 用ペイロードに変換。差分が空なら changed=false で終了。
API に書き込み 新規なら POST、既存なら PATCH（既定は Strategic Merge Patch、apply: true で Server-Side Apply）。失敗時は retries オプションで指数バックオフ。
結果を Ansible に返す レスポンス全体を result として返却。wait: true の場合はリソースが Ready 条件を満たすまで再 GET でポーリング。

認証・接続オプションの仕組み

認証情報の解決順序は 「モジュール引数 → 環境変数 → kubeconfig → in-cluster」です。 内部実装上は kubernetes クライアントの load_kube_config() / load_incluster_config() を呼び分けます。

方式指定方法内部で起きること
kubeconfig ファイル モジュール引数 kubeconfig: ~/.kube/config または環境変数 K8S_AUTH_KUBECONFIG。未指定なら標準パスを自動探索。 load_kube_config() が context / cluster / user を読み込み、CA・クライアント証明書・トークンを抽出。
明示トークン host, api_key, ca_cert を直接指定。 kubeconfig を完全に無視し、Configuration オブジェクトを手組み。CI/CD で Secret から渡すケースで便利。
In-cluster Pod 内で実行し incluster: true（または引数省略でフォールバック）。 /var/run/secrets/kubernetes.io/serviceaccount/ 配下のトークンと CA を読み、API Server を kubernetes.default.svc として参照。
OIDC / exec プラグイン kubeconfig 中の users.user.exec セクション（gcloud, aws-iam-authenticator など）。 クライアント側で外部コマンドを起動してトークンを取得 → リクエストヘッダに付与。EKS / GKE / AKS で頻出。
プロキシ proxy 引数または HTTPS_PROXY 環境変数。 urllib3 のプロキシハンドラ経由で kube-apiserver に到達。エアギャップ環境で重要。
注意

「kubectl get pods は通るのに Ansible からは通らない」場合、最初に疑うのは 制御ノードと Pod 内コンテナの kubeconfig パス違いと exec プラグインの PATH 欠落です。ansible -vvv でモジュール実行時の Python ログまで確認すると一発で切り分けられます。

冪等性とサーバーサイドアプライ

冪等性は Ansible モジュールの存在意義です。k8s モジュールは「現状と望ましい状態の差分」を見て初めて変更を発行するため、 同じ Playbook を 10 回流しても changed が立つのは差分があった最初の 1 回だけです。

① 現状（GET）

replicas: 2
image:
  nginx:1.25

② 望ましい状態

replicas: 3
image:
  nginx:1.25

③ 発行される PATCH

# 差分のみ送る
{ "spec": {
   "replicas": 3 }}

3 つの apply モード

モード挙動使いどころ
Strategic Merge Patch
（既定 / apply: false）		 Kubernetes が定義する戦略的マージ。containers リストなど patchMergeKey 付きフィールドは name で同一視されて差分計算される。 標準的な作成・更新。多くのケースでこれで十分。
Server-Side Apply
（apply: true）		 Kubernetes 1.22+ の SSA を使用。フィールド所有権を managedFields に記録し、複数の管理者（kubectl / Argo CD / Ansible）が同一リソースを上書きし合うのを防ぐ。 GitOps と並走させる場合、Operator と共存する場合に強く推奨。
force: true 競合（既存の所有権）を強制的に奪う。 意図的に上書きしたい場合のみ。常用すると SSA の利点が失われる。

kubernetes.core コレクションには k8s 以外にも目的別のモジュールが揃っています。

モジュール役割典型ユース

インベントリプラグインも同梱

kubernetes.core.k8s はインベントリプラグインも提供しており、Pod を Ansible の管理ホストとして自動列挙できます。k8s_exec と組み合わせると Pod 内へのプロビジョニングが Ansible の文法で書けます。

実例 Playbook

① 単純な Deployment / Service 作成

- hosts: localhost
  connection: local
  gather_facts: false
  tasks:
    - name: Ensure namespace exists
      kubernetes.core.k8s:
        state: present
        definition:
          apiVersion: v1
          kind: Namespace
          metadata:
            name: web

    - name: Deploy nginx
      kubernetes.core.k8s:
        state: present
        apply: true          # Server-Side Apply
        src: files/nginx-deploy.yaml
        wait: true
        wait_condition:
          type: Available
          status: "True"
        wait_timeout: 300

② Jinja2 テンプレートで動的生成

- name: Render and apply
  kubernetes.core.k8s:
    state: present
    template: templates/configmap.yaml.j2
  vars:
    app_name: payments
    log_level: debug

③ k8s_info で条件分岐

- name: Check if migration job finished
  kubernetes.core.k8s_info:
    api_version: batch/v1
    kind: Job
    name: db-migrate
    namespace: prod
  register: job

- name: Trigger only when previous job succeeded
  kubernetes.core.k8s:
    state: present
    src: files/app-deploy.yaml
  when: job.resources[0].status.succeeded | default(0) | int > 0

④ Pod 内コマンド実行（マイグレーション）

- name: Run schema migration inside app pod
  kubernetes.core.k8s_exec:
    namespace: prod
    pod: app-7d8f-xxxxx
    command: /app/manage.py migrate

kubectl apply との違い

kubectl apply
手続き的・スクリプトから呼ぶ前提
  • シェルから直接 / CI のコマンドステップとして実行
  • 差分判定は client-side か server-side apply で kubectl 自身が実施
  • 失敗時のリトライ・条件待ちは自前で書く必要
  • 変数展開やテンプレートは envsubst / Helm / Kustomize に頼る
  • 戻り値は exit code とテキスト出力のみ
kubernetes.core.k8s
宣言的・冪等・Ansible 全体に統合
  • Playbook の他タスク（OS 設定 / DB / 外部 API）と同じ実行コンテキスト
  • wait / wait_condition でリソース Ready まで待機を 1 行で表現
  • Jinja2 テンプレートや Ansible 変数とシームレスに連携
  • register したレスポンスを後続タスクで構造化された dict として参照
  • 失敗時の retries / until / rescue が標準利用可能

「単に YAML を適用するだけ」なら kubectl apply -f のほうがシンプルですが、「VM 構築 → Helm でクラスタ初期化 → アプリ Deploy → 監視設定 → 動作確認」のようなエンドツーエンドのプロビジョニングでは Ansible の表現力が活きます。

落とし穴とベストプラクティス

落とし穴 1

connection: ssh のまま k8s モジュールを実行すると、リモートノード上に Python の kubernetes クライアントが必要になります。多くの場合は delegate_to: localhost または connection: local にして制御ノードで実行するのが正解。

落とし穴 2

EKS / GKE で exec プラグイン（aws / gke-gcloud-auth-plugin）が PATH に無いと、kubectl は通っても Ansible の Python サブプロセス内で FileNotFoundError になります。ansible-playbook を実行する環境の PATH に必ず含めること。

落とし穴 3

Server-Side Apply を使わず Strategic Merge で書き続けると、Operator や Argo CD と同じフィールドを取り合って無限に changed が立つ「ピンポン現象」が起きることがあります。共存環境では apply: true を既定にしてフィールド所有権を分離するのが安全。

落とし穴 4

wait: true を付けずに Deployment を作成し、直後に Service / Ingress を作りに行くと、Pod が Ready になる前に外部からアクセスが来てしまうケースがあります。wait_condition で Available=True を待つのがベター。

Tip 1

大規模なマニフェストは src（既存 YAML ファイル）/ template（Jinja2）/ definition（インライン dict）のいずれかで指定します。変数展開を伴うなら template、手元の YAML をそのまま流すなら src、シンプルなものは definition、と使い分けると見通しが良くなります。

Tip 2

CI/CD では check_mode（ansible-playbook --check --diff）を活用すると、本番適用前に kube-apiserver が計算した差分を確認できます。diff は YAML 単位の見やすい形式で表示されます。

Tip 3

Secret や ConfigMap を Ansible 変数経由で渡す場合は Ansible Vaultで暗号化し、Playbook 内では no_log: true を付けてログ流出を防ぐと安全です。

応用：oc コマンドを k8s モジュールと同じように実装するなら

Red Hat OpenShift の oc は kubectl の上位互換 CLI で、Project / Route / BuildConfig / ImageStream / DeploymentConfig といった OpenShift 固有リソース、 そして oc new-app や oc login のような高レベルコマンドを持ちます。 これを kubernetes.core.k8s と同じ設計思想でモジュール化する場合、何を作り、何を流用すべきかを整理します。

先に結論

既に community.okd コレクションが同じ思想で実装済みで、内部的に kubernetes.core を継承して OpenShift API を扱います。自前実装するなら、その設計を再構築するのが正解です。以下はその根拠と内部構造の解説。

方針 1 — oc バイナリは使わない

kubernetes.core.k8s が kubectl を呼ばないのと同じく、oc バイナリも呼びません。 OpenShift の API Server は kube-apiserver + openshift-apiserver の集約 (aggregated API) 構成で、 どちらも同じエンドポイントから REST で叩けます。kubernetes Python クライアントの DynamicClient は API Discovery 時に両方のグループを自動で発見するため、Kubernetes リソースと OpenShift リソースを同じコードパスで扱えるのが鍵です。

OpenShift API Server の集約構造
openshift.core.oc モジュール（Python） 引数解決 / 認証 / 差分計算（k8s モジュールと同じ枠組み）
▼ kubernetes Python クライアント（DynamicClient）
kube-apiserver /api/v1, /apis/apps/v1, /apis/batch/v1 …
Pod / Deployment / Service / ConfigMap
openshift-apiserver /apis/project.openshift.io/v1
/apis/route.openshift.io/v1
/apis/build.openshift.io/v1 …
▼ aggregated API として 1 つのエンドポイントから提供
https://api.<cluster>:6443 クライアントから見ると単一の API Server に見える

方針 2 — OpenShift 固有リソースを表でマッピング

実装すべきリソースは oc サブコマンドから逆引きすると洗い出せます。 どれも「API グループ + Kind + 名前」さえ分かれば DynamicClient でそのまま CRUD でき、 冪等性ロジック（GET → diff → PATCH / SSA）は k8s モジュールと完全に同じものを使い回せます。

oc サブコマンド対応リソースAPI group特殊事情
oc new-projectProjectRequestproject.openshift.ioProject ではなく ProjectRequest を POST する（権限委譲のため）。
oc expose svcRouteroute.openshift.ioIngress と排他ではなく共存可。tls.termination を扱う必要あり。
oc new-buildBuildConfigbuild.openshift.ioSource / Strategy / Output の 3 セクション構成。
oc start-buildBuild（サブリソース）build.openshift.io/buildconfigs/<n>/instantiate という 非標準サブリソースを叩く。
oc tagImageStream / ImageStreamTagimage.openshift.ioタグ単位の更新は ImageStreamTag を PUT。
oc rolloutDeploymentConfigapps.openshift.ioレガシー（OpenShift 4.x では Deployment 推奨）。/instantiate サブリソースで手動ロールアウト。
oc adm policy add-scc-to-userSecurityContextConstraintssecurity.openshift.ioクラスタスコープ。RBAC との関係に注意。
oc whoamiUser / ~/users/meuser.openshift.io認証確認用。冪等性は不要。

方針 3 — oc login の OAuth フローを再現

oc login -u user -p pass の正体は OAuth2 の Implicit Grantです。 /oauth/authorize に response_type=token で BASIC 認証ヘッダを付けて GET すると、 302 リダイレクトの Location URL フラグメントに access_token=... が埋まって返ってきます。 これを抽出して以降のリクエストの Authorization: Bearer ヘッダに使うのが oc の動きです。 

# openshift.core.oc_login の内部実装イメージ（Python）
import requests, re, urllib.parse

def login(host, username, password, ca_cert):
    # 1. /.well-known/oauth-authorization-server で認可サーバを発見
    meta = requests.get(f"{host}/.well-known/oauth-authorization-server",
                        verify=ca_cert).json()
    auth_url = meta["authorization_endpoint"]

    # 2. challenging-client として Implicit Grant をリクエスト
    params = {
        "response_type": "token",
        "client_id":     "openshift-challenging-client",
    }
    r = requests.get(auth_url, params=params,
                     auth=(username, password),
                     headers={"X-CSRF-Token": "1"},
                     allow_redirects=False, verify=ca_cert)

    # 3. Location ヘッダのフラグメントから access_token を抽出
    fragment = urllib.parse.urlparse(r.headers["Location"]).fragment
    token = urllib.parse.parse_qs(fragment)["access_token"][0]

    # 4. 以降は k8s モジュールと同じ Bearer Token として使う
    return token

取得したトークンは K8S_AUTH_API_KEY 環境変数 / モジュール引数 api_key としてそのまま kubernetes.core.k8s に渡せます。「oc_login で token を取り、以降は k8s モジュールで CRUD」 という二段構成が最小実装です。

方針 4 — 高レベルコマンドは「複合タスク」に分解

oc new-app nginx は 1 つの API ではありません。サーバー側で以下の 4〜5 リソースを まとめて作成しています。モジュール側で同じ分解をして、それぞれを冪等に処理すれば再現できます。

ImageStream を作成 イメージの参照を OpenShift 内部レジストリに登録（image.openshift.io/v1/imagestreams）。
BuildConfig を作成（Git ソース指定時のみ） S2I ビルドの定義を build.openshift.io/v1/buildconfigs に POST。
Deployment / DeploymentConfig を作成 標準の apps/v1 Deployment を使うのが現代流。
Service を作成 標準の v1 Service。Pod のラベルセレクタを合わせる。
Route を作成（--expose 時） route.openshift.io/v1/routes を POST。HAProxy ルーターが拾って外部公開。

各ステップを kubernetes.core.k8s 相当のヘルパー関数で実装し、 1 つでも変更があれば全体の changed=true を返す集計ロジックを足せば、 oc new-app と同じ「高レベル冪等タスク」が完成します。

提案するモジュール構成

モジュール役割k8s 側で十分か
openshift.core.openshift低レベルの REST ラッパー（k8s の OpenShift 版）不要。kubernetes.core.k8s が OpenShift リソースもそのまま扱える。薄いエイリアスのみで OK。
openshift.core.openshift_authoc login 相当の OAuth トークン取得必要。kubernetes クライアントは Implicit Grant を実装していない。
openshift.core.openshift_processoc process 相当の Template 展開必要。Template の Parameter 注入を /processedtemplates サブリソースに POST。
openshift.core.openshift_routeRoute の便利モジュール（TLS / 証明書 / wildcard policy）薄いラッパーでよい。本体は k8s モジュールで作れるが、TLS 設定の構造を引数化すると使いやすい。
openshift.core.openshift_buildoc start-build 相当（/instantiate サブリソース）必要。標準 CRUD ではないため k8s モジュールでは扱いにくい。
openshift.core.openshift_import_imageoc import-image（外部レジストリの取り込み）必要。ImageStreamImport という即時実行リソースを POST する特殊仕様。
openshift.core.openshift_adm_policySCC / Role の付与ユーティリティあれば便利。RoleBinding を直接書くより読みやすい。

設計の要点まとめ

① 再利用最大化

標準 CRUD は kubernetes.core.k8s に丸投げ。OpenShift 専用ロジックだけ薄く足す。

② 認証だけは別実装

oc login の OAuth Implicit Grant は kubernetes クライアントが持っていないので、新規モジュールで実装する。

③ サブリソース系は専用モジュール

/instantiate / /processedtemplates / imagestreamimports など、CRUD で表現できないものは個別モジュール化。

④ 高レベルは分解して冪等化

oc new-app 系は内部で複数リソース作成。1 モジュール内で順序立てて作成し、全体 changed を集約。

補足

community.okd はまさにこの方針で書かれており、openshift_auth / openshift_route / openshift_process / openshift_import_image / openshift_adm_prune_deployments といったモジュールを提供しつつ、CRUD は kubernetes.core.k8s を継承しています。「足りないところだけ足す」設計の好例として一読の価値があります。
