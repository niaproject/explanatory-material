Google Workspace CLI (gws) 入門

27 サービスをターミナル 1 つで操る、Google 謹製の CLI

npm i -g @googleworkspace/cli

これは何？ — 正体をひとことで

Google Workspace CLI（通称 gws）は、Gmail / Calendar / Drive / Docs / Sheets / Slides など 27 の Google サービスをコマンドラインから操作できる CLI ツールです。 Google の Developer Relations チームが開発し、googleworkspace 公式 GitHub 組織で配布しています。

パッケージ名@googleworkspace/cli
実行コマンドgws
リポジトリgithub.com/googleworkspace/cli
配布npm（グローバルインストール）
実装言語Node.js / TypeScript
バージョン例0.3.4 時点
重要な注意: README には "This is not an officially supported Google product." と明記されています。Google 謹製ではあるが公式サポート対象外の位置づけです。 本番運用で使う場合は、この前提を踏まえてバージョンを固定し、自分で再現手順を管理してください。

共通コマンド文法

すべてのサービスに対して、同じ文法で呼べるのが gws の最大の特徴です。

gws <service> <resource> [subresource] <method> [flags]
service
対象サービス名（docs, drive, gmail …）
resource
API リソース（documents, files, messages …）
subresource
ネストされたリソース（省略可）
method
API メソッド（list / get / create / batchUpdate …）

主要なフラグ

フラグ用途
--params <JSON>URL パス / クエリパラメータ（documentId などはここ）
--json <JSON>リクエストボディ（batchUpdate の requests はここ）
--format <fmt>出力形式: json / table / yaml / csv
--upload <path>ファイルアップロード時のローカルパス
--page-allページネーションを自動で全ページ巡回

特徴

🧩
API Discovery 自動生成
Google API の Discovery Document からサブコマンドを自動生成するため、公式 API で呼べるメソッドはほぼそのまま使える。
📦
構造化出力
レスポンスは JSON / YAML / CSV / table に切り替え可能。jq やシェルパイプとの相性が良い。
🔐
OAuth ログイン内蔵
gws auth login でブラウザ認可 → 資格情報をローカルに暗号化保存。サービスアカウント運用も可。
📑
スキーマ確認コマンド
gws schema <service>.<method> でリクエスト / レスポンスの型をその場で確認できる。

インストール

Node.js が入っていれば、npm 1 コマンドで導入完了です。

npm install -g @googleworkspace/cli
gws --version      # 例: 0.3.4
前提: Node.js は LTS（18 以降）推奨。macOS / Linux なら nvm、Windows なら公式インストーラまたは winget install OpenJS.NodeJS.LTS が手軽です。

初回セットアップ（OAuth 認可）

gws は対象ユーザーとして Google の API を呼ぶため、OAuth クライアント（デスクトップアプリ）の発行と認可が必要です。 推奨ルートと、gcloud が使えない場合の手動ルートの 2 つがあります。

推奨: gws auth setup を使う

Google Cloud プロジェクトの作成・API 有効化・OAuth クライアントの発行までを gws 自身が面倒を見てくれるモードです。 事前に gcloud CLI がインストール済みで、gcloud auth login も済んでいる必要があります。 

gws auth setup    # GCP プロジェクト作成 + API 有効化 + OAuth クライアント発行
gws auth login    # 使いたいスコープを選んでブラウザで同意

手動: Google Cloud Console から作る

gcloud を使えない / 使いたくない場合はこちら。

Google Cloud Console でプロジェクトを用意

console.cloud.google.com でプロジェクトを選択（または新規作成）。

OAuth 同意画面を作る

User Type: External を選び、テストモードのまま、テストユーザーに自分の Google アカウントを追加します（追加を忘れると認可で弾かれる）。

OAuth クライアントを発行

Cloud Console 左メニューの 「API とサービス」 → 「認証情報」 に移動します。

≡ › API とサービス › 認証情報

ページ上部の「+ 認証情報を作成」→「OAuth クライアント ID」を選択。 アプリケーションの種類で デスクトップ アプリ (Desktop app) を選び、名前を付けて「作成」を押します。

発行されたクライアントの行にある JSON をダウンロード ボタンから認証情報ファイルを取得し、 ~/.config/gws/client_secret.json に配置します。

直接 URL: メニューから辿れない場合は https://console.cloud.google.com/apis/credentials?project=PROJECT_ID を開けば「認証情報」ページに直接行けます。

必要な API を有効化

使うサービスの API を Cloud Console から ON にします。gcloud があるなら一括で：

gcloud services enable \
  docs.googleapis.com \
  drive.googleapis.com \
  sheets.googleapis.com \
  gmail.googleapis.com \
  calendar-json.googleapis.com \
  slides.googleapis.com

ログインしてスコープを選ぶ

gws auth login -s docs,drive,sheets

ブラウザが開き、Google アカウントで同意すると資格情報がローカルに保存されます。

スコープ数の上限: テストモード（未検証アプリ）では 1 アプリにつき 約 25 スコープまでという制限があります。使うサービスのスコープだけを -s で指定してください。

認証情報の優先順位

優先度ソース環境変数 / ファイル
1アクセストークン直指定GOOGLE_WORKSPACE_CLI_TOKEN
2資格情報ファイル指定GOOGLE_WORKSPACE_CLI_CREDENTIALS_FILE
3暗号化保存済みトークンgws auth login が保存したもの

gcloud services で API を管理する

gws は裏で Google の各種 API（docs.googleapis.com / drive.googleapis.com など）を叩いています。 これらは使う前に GCP プロジェクト単位で「有効化」しておく必要があり、 その有効化 / 無効化 / 確認を一括で行えるのが gcloud services サブコマンドです。

用語の整理: Google Cloud 上で「API」と呼ばれるものは内部的に service という単位で管理されています。 だから有効化コマンドは gcloud apis … ではなく gcloud services … です。

API の有効化: gcloud services enable

指定したプロジェクトで API を有効化します。スペース区切りで複数同時有効化が可能。

# 単一 API を有効化（現在アクティブなプロジェクトに対して）
gcloud services enable docs.googleapis.com

# 複数 API を一括で有効化（Workspace CLI でよく使う組み合わせ）
gcloud services enable \
  docs.googleapis.com \
  drive.googleapis.com \
  sheets.googleapis.com \
  gmail.googleapis.com \
  calendar-json.googleapis.com \
  slides.googleapis.com

# プロジェクトを明示指定
gcloud services enable docs.googleapis.com --project=PROJECT_ID
有効化には数十秒かかる: 裏側で Service Usage API に非同期ジョブが走るため、 gcloud services enable が完了した直後に API を叩くと PERMISSION_DENIED / SERVICE_DISABLED が返ることがあります。 少し待つか、次節の services list --enabled で実際に有効化されたか確認してから実行してください。

有効化されている API の確認: gcloud services list --enabled

現在のプロジェクトですでに有効化されている API 一覧を表示します。

# 有効化済みの API 一覧
gcloud services list --enabled

# 例: 出力（抜粋）
# NAME                                 TITLE
# docs.googleapis.com                  Google Docs API
# drive.googleapis.com                 Google Drive API
# sheets.googleapis.com                Google Sheets API

# 特定の API が有効か素早く調べる（grep）
gcloud services list --enabled | grep docs

# Workspace 関連だけ抽出（PowerShell の場合）
gcloud services list --enabled | Select-String 'docs|drive|sheets|gmail|calendar|slides'

# CSV / JSON で取り出す（スクリプトで扱いやすい）
gcloud services list --enabled --format="value(config.name)"
gcloud services list --enabled --format=json

すべての API（有効化可能なもの）を確認: gcloud services list --available

そのプロジェクトで有効化可能な全 APIの一覧です。数が多いので絞り込みが前提になります。

# 全 API 一覧（数百件出るので注意）
gcloud services list --available

# 「docs」を含む API だけに絞る（--filter 推奨）
gcloud services list --available --filter="name:docs"

# Workspace 系 API をまとめて検索
gcloud services list --available \
  --filter="name:(docs OR drive OR sheets OR gmail OR calendar OR slides)"

# タイトルで検索（表示名に「Workspace」が入るもの）
gcloud services list --available --filter="title:Workspace"
--enabled と --available の違い:
  • --enabled: すでに有効化されている API（＝このプロジェクトで呼び出せる API）
  • --available: このプロジェクトで有効化できる API 全体（組織ポリシーで制限されていないもの）

API の無効化: gcloud services disable

使わなくなった API を無効化します。不要な API が有効なままだと攻撃面の広がりやクォータ管理の煩雑化につながるので、整理したいときに。

# 単一 API を無効化
gcloud services disable docs.googleapis.com

# 依存されていても強制的に無効化（推移的に依存 API も止まる）
gcloud services disable docs.googleapis.com --force

# プロジェクトを明示指定
gcloud services disable docs.googleapis.com --project=PROJECT_ID
⚠ 無効化の影響:
  • 無効化した瞬間から、その API へのリクエストはすべて 403 SERVICE_DISABLED
  • 他の有効な API から依存されている場合、--force を付けないと無効化は失敗する
  • --force を付けると依存 API も止まり、連鎖的に壊れる可能性がある — 本番プロジェクトでは事前に依存を確認

よく使うフラグ一覧

フラグ意味使いどころ
--project=ID 対象プロジェクトを指定 複数プロジェクトを触っているとき。指定しないと gcloud config のデフォルトが使われる
--filter="name:…" 名前で絞り込み --available の結果を grep 相当で絞るときの標準手段
--format=… 出力形式を指定 value(config.name) / json / yaml / csv。スクリプトで使うなら必須
--force 依存を無視して無効化 disable 専用。依存エラーを回避したいときだけ使う
--async 非同期実行 enable / disable 時に待たずに即座に制御を戻す

すぐ使えるコマンド例

gws drive files list --params '{"pageSize": 5}'

gws gmail users messages list --params '{"userId": "me"}'

gws people people get --params '{"resourceName": "people/me", "personFields": "names,emailAddresses"}'

方法 A: ヘルパー gws docs +write

gws には Docs 向けに 内蔵ヘルパーが用意されています。+ プレフィックスの付くコマンドは、生の API を呼び出すのではなく、よく使う操作を 1 コマンドにまとめたものです。 単純に「本文にテキストを追記したい」だけならこれが最短です。 

gws docs +write --help   # 受け付けるオプションを確認

方法 B: documents.batchUpdate を直接呼ぶ

本格的な編集は、Docs API の batchUpdate エンドポイントに リクエスト配列を渡します。 gws では --params に documentId、--json に { "requests": [...] } を渡す形になります。

冒頭にテキストを挿入

gws docs documents batchUpdate \
  --params '{"documentId": "YOUR_DOC_ID"}' \
  --json '{
    "requests": [
      {
        "insertText": {
          "location": { "index": 1 },
          "text":     "2026-04-18 自動追記: ビルド成功\n"
        }
      }
    ]
  }'

テンプレ変数を一括置換

gws docs documents batchUpdate \
  --params '{"documentId": "YOUR_DOC_ID"}' \
  --json '{
    "requests": [
      {"replaceAllText": {
        "containsText": {"text": "{{client}}", "matchCase": true},
        "replaceText":  "株式会社サンプル"
      }},
      {"replaceAllText": {
        "containsText": {"text": "{{date}}", "matchCase": true},
        "replaceText":  "2026-04-18"
      }}
    ]
  }'

ドキュメントの現在の中身を取得

gws docs documents get --params '{"documentId": "YOUR_DOC_ID"}' --format json

空のドキュメントを作成

gws docs documents create --json '{"title": "議事録 2026-04"}'

よく使う batchUpdate リクエスト種別

リクエスト用途
insertText指定 index にテキストを挿入
deleteContentRange範囲を削除
replaceAllText文字列の一括置換（テンプレ差し込み）
insertTable表を挿入
insertInlineImage画像を本文に埋め込み
updateParagraphStyle見出しレベル・配置・行間
updateTextStyle太字 / 斜体 / リンク / 色
createParagraphBullets箇条書きにする
スキーマ確認: 使えるリクエストが多くて迷うときは gws schema docs.documents.batchUpdate でリクエストの型定義をその場で吐き出せます。公式リファレンスを開かずに CLI 内で完結できるのが gws の便利なところです。

実運用レシピ

テンプレ差し込みで契約書を量産

  1. gws drive files copy でテンプレ Doc を複製して fileId を得る
  2. gws docs documents batchUpdate + replaceAllText で {{client}} / {{date}} 等を埋める
  3. gws drive permissions create で先方メールに閲覧権限を付与

CI から日次レポートを書き込む

  1. サービスアカウントを発行し、対象 Doc にそのメールアドレスで編集権を付与
  2. 環境変数 GOOGLE_WORKSPACE_CLI_CREDENTIALS_FILE にサービスアカウント鍵パスを指定
  3. CI で gws docs documents batchUpdate を叩く
鍵の扱い: OAuth トークン / サービスアカウント鍵は「対象ドキュメントの編集権」そのものです。 ~/.config/gws/ や鍵 JSON を絶対に Git にコミットしない、CI では秘匿変数に入れる、を徹底してください。

まとめ

  • 「Google Workspace CLI」とは @googleworkspace/cli（gws）のこと。Google 謹製だが公式サポート外。
  • インストールは npm install -g @googleworkspace/cli の 1 行、初期化は gws auth setup → gws auth login。
  • 全サービスで共通の文法 gws <service> <resource> <method> --params ... --json ... で呼べる。
  • Google ドキュメントの編集は gws docs +write（簡易）か gws docs documents batchUpdate（本格）。
  • replaceAllText によるテンプレ差し込みが、もっとも実用度が高い定番パターン。
