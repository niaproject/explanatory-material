gcloud CLI のインストール方法

Windows / macOS / Linux 別の推奨手順と、初期セットアップ・更新・アンインストールまで

gcloud CLI とは

gcloud CLI（旧 Cloud SDK）は、Google Cloud のリソースをターミナルから操作するための公式 CLI ツールです。 以下のツール群が同梱されています。

  • gcloud — Compute Engine / GKE / IAM / プロジェクト管理などメイン CLI
  • gsutil — Cloud Storage（バケット / オブジェクト）操作
  • bq — BigQuery 操作
  • gcloud storage — gsutil の後継。大容量転送が高速
  • オプションで kubectl / minikube / skaffold などのコンポーネントを追加可能
前提: Python 3.9 以上が同梱されるため、別途 Python を用意する必要はありません（古いバンドルでは Python 3.8 同梱のこともあります）。 ネット接続、ディスク空き容量 約 500 MB が必要です。

OS ごとの推奨インストール方法

まずは早見表。迷ったらここのおすすめで入れれば OK です。

🪟Windows
推奨: インストーラ (.exe)
公式の GoogleCloudSDKInstaller.exe を実行。PATH 登録も自動。
🍎macOS
推奨: アーカイブ or Homebrew
Apple Silicon / Intel 別の .tar.gz を展開。Homebrew 派は brew install --cask google-cloud-sdk。
🐧Linux (Debian / Ubuntu)
推奨: apt リポジトリ
公式 apt リポジトリを追加して apt install google-cloud-cli。更新が楽。
🐧Linux (RHEL / Fedora)
推奨: dnf/yum リポジトリ
/etc/yum.repos.d/google-cloud-sdk.repo を作成して dnf install google-cloud-cli。

OS 別インストール手順

公式インストーラ (.exe) 推奨
PATH 登録・Python 同梱・初回 gcloud init 起動までワンストップ。迷ったらこれ。

インストーラをダウンロード

公式ダウンロードページから GoogleCloudSDKInstaller.exe を取得します。

https://cloud.google.com/sdk/docs/install#windows

インストーラを実行

ダブルクリックで起動し、ウィザードに従って進めます。ユーザー単位 / マシン全体どちらのインストールかを選択できます。

  • 「Single user」: 自分のユーザーにのみ適用（管理者権限不要）
  • 「All users」: システム全体に適用（管理者権限が必要）

完了後、PowerShell を開き直す

PATH を反映させるため、今開いている PowerShell / コマンドプロンプトは一度閉じて開き直します。

動作確認と初期化

gcloud version
gcloud init

gcloud init を実行するとブラウザが開き、Google アカウントでログインを求められます。

PowerShell 実行ポリシー: 組織の GPO で Restricted になっているとインストーラの一部スクリプトが失敗することがあります。 その場合は管理者 PowerShell で Set-ExecutionPolicy RemoteSigned -Scope CurrentUser を実行してから再試行。
Chocolatey 選択肢
パッケージ管理で入れたい人向け。更新も choco upgrade 一発。
# 管理者 PowerShell で
choco install gcloudsdk

# 更新
choco upgrade gcloudsdk
Scoop 選択肢
管理者権限不要。Scoop ユーザーならこちら。
scoop bucket add extras
scoop install gcloud
手動アーカイブ展開 上級
オフライン環境用。google-cloud-cli-windows-x86_64.zip を任意フォルダに展開し、同梱の install.bat を実行して PATH を通します。
PATH を手動で通す（PowerShell） トラブルシュート
インストーラで入れたのに gcloud: command not found が出る、または手動アーカイブ展開後に PATH が通っていない場合、 PowerShell から ユーザー環境変数の PATH に直接追記できます。 
# ユーザー環境変数 PATH に gcloud の bin を追記
[Environment]::SetEnvironmentVariable("Path", $env:Path + ";$env:LOCALAPPDATA\Google\Cloud SDK\google-cloud-sdk\bin", "User")

このコマンドが何をしているか

  • [Environment]::SetEnvironmentVariable(名前, 値, スコープ) で環境変数を永続的に書き換える .NET API 呼び出し
  • 第 1 引数 "Path": 書き換える変数名
  • 第 2 引数 $env:Path + ";…\bin": 現在の PATH の末尾に gcloud の bin フォルダを追加
  • 第 3 引数 "User": ユーザー環境変数に保存（"Machine" にすればシステム全体／管理者権限必須）
  • $env:LOCALAPPDATA は通常 C:\Users\<ユーザー名>\AppData\Local に展開される

確認手順

# 1. 現在開いている PowerShell をいったん閉じて、新しく開き直す
#    （SetEnvironmentVariable の結果は既存セッションには反映されない）

# 2. PATH に追加されたか確認
$env:Path -split ';' | Select-String 'Cloud SDK'

# 3. gcloud が呼べるか確認
gcloud version
ハマりポイント:
  • 現在のシェルには反映されない — 新しい PowerShell / コマンドプロンプトを開き直す必要があります。
  • 重複追加に注意 — 複数回実行すると同じパスがいくつも並びます。実行前に $env:Path に含まれていないか確認を。
  • インストール先が違う場合 — 「All users」でインストールしていると実体は C:\Program Files (x86)\Google\Cloud SDK\google-cloud-sdk\bin にあるので、第 2 引数のパスを差し替えます。
「Machine」スコープで入れたい場合: 管理者 PowerShell を開き、第 3 引数を "Machine" に。 ただしユーザー単位で完結させられるならユーザースコープのほうが安全です。 
# 管理者 PowerShell で
[Environment]::SetEnvironmentVariable("Path", $env:Path + ";C:\Program Files (x86)\Google\Cloud SDK\google-cloud-sdk\bin", "Machine")
公式アーカイブ (.tar.gz) 推奨
Apple Silicon (ARM64) / Intel (x86_64) 別のアーカイブがあります。間違えると起動時に Rosetta を要求されます。

自分の Mac のアーキテクチャを確認

uname -m
# → arm64  なら Apple Silicon (M1/M2/M3)
# → x86_64 なら Intel Mac

アーキテクチャに合ったアーカイブを取得

# Apple Silicon (arm64)
curl -O https://dl.google.com/dl/cloudsdk/channels/rapid/downloads/google-cloud-cli-darwin-arm.tar.gz

# Intel (x86_64)
curl -O https://dl.google.com/dl/cloudsdk/channels/rapid/downloads/google-cloud-cli-darwin-x86_64.tar.gz

展開してインストーラを実行

tar -xf google-cloud-cli-darwin-arm.tar.gz
./google-cloud-sdk/install.sh

対話形式で PATH 追加・コマンド補完の設定を聞かれます。基本は全部 Y で OK。

シェルを再起動して初期化

exec -l $SHELL
gcloud init
Homebrew (Cask) 選択肢
Mac の定番パッケージマネージャ。brew upgrade で更新できて楽。
brew install --cask google-cloud-sdk

# PATH / 補完を有効化（zsh の場合）
echo 'source "$(brew --prefix)/share/google-cloud-sdk/path.zsh.inc"' >> ~/.zshrc
echo 'source "$(brew --prefix)/share/google-cloud-sdk/completion.zsh.inc"' >> ~/.zshrc
source ~/.zshrc
ハマりポイント: Cask 版は brew install google-cloud-sdk（Formula 版）ではなく brew install --cask google-cloud-sdk です。Formula 版は存在しません。
Debian / Ubuntu — apt リポジトリ 推奨
公式の apt リポジトリを追加する方式。apt upgrade で自動更新されます。
# 1. 必要なパッケージを導入
sudo apt-get update
sudo apt-get install apt-transport-https ca-certificates gnupg curl

# 2. Google の公開鍵を登録
curl https://packages.cloud.google.com/apt/doc/apt-key.gpg \
  | sudo gpg --dearmor -o /usr/share/keyrings/cloud.google.gpg

# 3. リポジトリを追加
echo "deb [signed-by=/usr/share/keyrings/cloud.google.gpg] \
  https://packages.cloud.google.com/apt cloud-sdk main" \
  | sudo tee /etc/apt/sources.list.d/google-cloud-sdk.list

# 4. インストール
sudo apt-get update
sudo apt-get install google-cloud-cli

# 5. 初期化
gcloud init
kubectl や GKE 認証プラグインも入れるなら: 
sudo apt-get install google-cloud-cli-gke-gcloud-auth-plugin kubectl
RHEL / Fedora / Rocky — dnf リポジトリ 推奨
Red Hat 系はリポジトリファイルを作成してから dnf install。
# 1. リポジトリファイルを作成
sudo tee /etc/yum.repos.d/google-cloud-sdk.repo <<EOF
[google-cloud-cli]
name=Google Cloud CLI
baseurl=https://packages.cloud.google.com/yum/repos/cloud-sdk-el9-x86_64
enabled=1
gpgcheck=1
repo_gpgcheck=0
gpgkey=https://packages.cloud.google.com/yum/doc/rpm-package-key.gpg
EOF

# 2. インストール
sudo dnf install google-cloud-cli

# 3. 初期化
gcloud init
OS バージョンに合わせる: cloud-sdk-el9-x86_64 の el9 は RHEL 9 / Rocky Linux 9 用。RHEL 8 なら el8、ARM 系なら -aarch64 に差し替えます。
Snap 選択肢
snap が使える環境（Ubuntu など）なら 1 行で入ります。ただし更新頻度が遅めなのがデメリット。
sudo snap install google-cloud-cli --classic
手動アーカイブ (.tar.gz) 選択肢
root 権限のない環境や、特定バージョンに固定したい CI で使う方式。
# ダウンロード (x86_64 の例)
curl -O https://dl.google.com/dl/cloudsdk/channels/rapid/downloads/google-cloud-cli-linux-x86_64.tar.gz

# 展開とインストール
tar -xf google-cloud-cli-linux-x86_64.tar.gz
./google-cloud-sdk/install.sh
exec -l $SHELL
gcloud init
CI での使い方: 本番の Dockerfile やパイプラインで使うなら、公式イメージ gcr.io/google.com/cloudsdktool/google-cloud-cli:stable を使うのが一番確実です。

インストール後の初期セットアップ

インストールが終わったら、次の 3 ステップでアカウントとプロジェクトを結びつけます。

gcloud init — 対話形式でまとめて設定

初回はこれを叩けば OK。ブラウザが自動で開き、Google アカウントでの認可→デフォルトプロジェクト・リージョン選択まで一気に終わります。

gcloud init

Application Default Credentials (ADC) の設定

ローカルで走らせる Python / Node.js 等の SDK（BigQuery クライアントなど）は、 gcloud auth login とは別の ADC 認証を使います。 

gcloud auth application-default login
混同注意: gcloud auth login は gcloud 自身用、 gcloud auth application-default login はクライアントライブラリ用。 「ADC が見つからない」というエラーが出たら後者を実行するのが定番の解決策です。

現在の設定を確認

gcloud config list
gcloud auth list
gcloud projects list

アカウント・プロジェクト・リージョンが想定どおりか確認しておきます。

gcloud init を実行すると何が起きるか

gcloud init は「gcloud CLI と Google アカウント・プロジェクトを紐付ける対話セットアップ」を行うコマンドです。 内部的には gcloud auth login + gcloud config set project … + gcloud config set compute/zone … を順に実行するラッパーと考えると分かりやすいです。

実行フロー

設定プロファイル（configuration）の選択

既存の configuration を使うか、新規作成するかを聞かれます。 configuration は「アカウント・プロジェクト・リージョンの組」の名前付きセットで、保存先は下記。

  • Linux / macOS: ~/.config/gcloud/configurations/config_<name>
  • Windows: %APPDATA%\gcloud\configurations\config_<name>
使いどころ: 複数案件を切り替えるなら案件ごとに configuration を分けると安全。 gcloud config configurations activate <name> で切替、gcloud config configurations list で一覧表示。

アカウント認証（OAuth ログイン）

gcloud auth login 相当の処理が走ります。 ブラウザが自動で開き、Google のログイン画面と「Google Cloud SDK がアカウントへのアクセスをリクエストしています」の同意画面が表示されます。 許可すると OAuth refresh token が ~/.config/gcloud/ に保存されます。

これは gcloud 自身用の認証: アプリ（Python / Node.js など）のクライアントライブラリが使う ADC とは別物です。ADC は別途 gcloud auth application-default login が必要。

デフォルトプロジェクトの選択

認可したアカウントがアクセスできるプロジェクト一覧が表示され、番号を入力して選びます。 新規プロジェクトをその場で作成することも可能。選んだ ID は core/project に保存され、以降 --project 省略で効きます。

Compute Engine 既定リージョン / ゾーンの設定（任意）

Compute Engine API が有効なプロジェクトだと、デフォルトゾーンを聞かれます。 asia-northeast1-a 等を選ぶと compute/zone と compute/region に保存されます。

「Do you want to configure a default Compute Region and Zone?」で N を押せばスキップ可能。 Compute Engine を使わないなら不要です。

結果として保存されるもの

実行後、~/.config/gcloud/（Windows は %APPDATA%\gcloud\）配下に以下のファイル群が作られます。

~/.config/gcloud/
├── configurations/
│   └── config_default         # [core] account, project / [compute] zone, region
├── credentials.db             # OAuth refresh token
├── access_tokens.db           # 短命アクセストークンキャッシュ
└── active_config              # 現在アクティブな configuration 名

完了後すぐに効く状態

gcloud config list              # 保存された内容を確認
gcloud projects list            # 権限のあるプロジェクト一覧
gcloud compute instances list   # デフォルトプロジェクト/ゾーンで実行される

注意点

ポイント内容
ADC は別扱い Python / Node などのクライアントライブラリ用の認証は gcloud auth application-default login を別に実行しないと効かない。
ブラウザが無い環境 SSH 先 / CI などでは gcloud init --no-browser（古い版は --console-only）で、URL と認証コードを手コピーする方式に切替可能。
サービスアカウント運用 init は不要。gcloud auth activate-service-account --key-file=… + gcloud config set project … で直接設定するのが適切。
やり直したい 複数回実行しても壊れない。gcloud config configurations create <new> で別 configuration を作り直せばクリーンにやり直せる。
まとめると: gcloud init は「OAuth ログイン + デフォルトプロジェクト設定を対話で済ませるためのラッパー」。 内部で行っていることを個別コマンドで書くとこう ↓ 
gcloud auth login
gcloud config set project PROJECT_ID
gcloud config set compute/region asia-northeast1
gcloud config set compute/zone   asia-northeast1-a

コンポーネントの追加（kubectl など）

gcloud には「コンポーネント」という追加モジュールの仕組みがあります。 使うツールだけ選んで追加できるので、無駄に肥大化しません。

利用可能なコンポーネント一覧

gcloud components list

よく使うコンポーネントの追加

# kubectl（GKE を使うなら必須）
gcloud components install kubectl

# GKE 認証プラグイン（kubectl 1.26+ では必須）
gcloud components install gke-gcloud-auth-plugin

# beta / alpha チャンネル
gcloud components install beta alpha

# minikube / skaffold / Cloud SQL Auth proxy など
gcloud components install minikube skaffold cloud-sql-proxy
apt / dnf で入れた場合: gcloud components install が使えません（パッケージ管理の責務を奪わないため）。 該当コンポーネントは google-cloud-cli-kubectl のような個別パッケージで導入します。 
sudo apt-get install google-cloud-cli-gke-gcloud-auth-plugin kubectl

アンインストール

# Windows: コントロールパネルの「プログラムのアンインストール」から
#          または C:\Users\<user>\AppData\Local\Google\Cloud SDK\uninstaller.exe

# macOS / Linux (アーカイブ展開版): 展開フォルダごと削除
rm -rf ~/google-cloud-sdk
rm -rf ~/.config/gcloud

# Homebrew
brew uninstall --cask google-cloud-sdk

# apt / dnf
sudo apt-get remove google-cloud-cli
sudo dnf remove google-cloud-cli
認証情報も消すには: ~/.config/gcloud/ (Linux/Mac) / %APPDATA%\gcloud\ (Windows) を削除します。ここに ADC トークンや設定 (configuration) が入っています。

よくあるハマりどころ

症状原因対処
gcloud: command not found PATH が通っていない / シェルを再起動していない 新しいターミナルを開く。macOS/Linux は ~/.zshrc に path.zsh.inc の source を追記。Windows は PowerShell の [Environment]::SetEnvironmentVariable("Path", …, "User") で PATH を手動追加
ADC エラー（Could not automatically determine credentials） ADC がまだ発行されていない gcloud auth application-default login
kubectl が GKE に繋がらない（exec plugin is missing） GKE 認証プラグイン未導入 gcloud components install gke-gcloud-auth-plugin または個別パッケージ
gcloud components update が失敗 apt / dnf 管理下では無効 パッケージマネージャでアップデート
Mac で「開発元を確認できない」警告 macOS Gatekeeper システム設定 → プライバシーとセキュリティで「このまま開く」を許可
プロキシ環境で接続できない プロキシ未設定 gcloud config set proxy/type http などで設定 or HTTPS_PROXY 環境変数

インストール後チェックリスト

最初の検証として、以下を一通り実行できれば完了です。

  • gcloud version でバージョンが表示される
  • gcloud init で Google アカウントとプロジェクトを紐付けた
  • gcloud auth application-default login で ADC を発行した
  • gcloud config list で account / project / region が想定どおり
  • gcloud projects list で自分が権限を持つプロジェクトが出る
  • シェル補完（タブキーでコマンド補完）が効く
  • （GKE を使うなら）kubectl と gke-gcloud-auth-plugin が入っている

まとめ

  • Windows は公式インストーラ、macOS はアーカイブ or Homebrew Cask、Linux はapt / dnf リポジトリが推奨。
  • インストール後の定型セットは gcloud init → gcloud auth application-default login の 2 本立て。
  • 追加ツールは gcloud components install（アーカイブ版）か、パッケージ版の個別サブパッケージで追加。
  • 更新はインストール方法ごとにコマンドが違うので、入れた経路に合わせて揃えるのが鉄則。
  • CI や Docker で使うなら、公式イメージ google-cloud-cli:stable を使うのが一番安定します。
