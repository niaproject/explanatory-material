GitHub アカウント情報はどこに保存されている？

VSCode 側 vs OS 側（Windows 資格情報マネージャー / macOS キーチェーン）の二層構造を徹底整理

結論（4 行まとめ）

  • GitHub の認証情報は 1 箇所ではなく複数の経路に分散保存されている（VSCode / git CLI / gh CLI / SSH 鍵）。
  • VSCode の「GitHub サインイン」で入れたトークンは VSCode の SecretStorage に保存される（ただし物理的には OS キーチェーンを使う）。
  • git push / git pull で使う認証情報は、VSCode とは別経路で Git Credential Manager 経由で OS キーチェーンに直接保存される。
  • 実体ファイルは OS 側（Windows: 資格情報マネージャー / macOS: login.keychain）、エントリ名や呼び出し口が VSCode 側 or git 側で分かれている、という構造。

全体像 — 3 層の保存構造

「GitHub の認証情報がどこに保存されるか」を考えるときは、3 つの階層に分けて見るとスッキリします。 上から「誰が使うか（クライアント）」「誰が保管役を引き受けるか（ストレージ API）」「どこに物理的に保存するか（OS 基盤）」です。

① Clients
認証情報を使う側（どこから呼び出されるか）
GitHub へ接続するあらゆるアプリ。それぞれが別経路で認証情報を取りに行く。
VSCode GitHub 拡張 GitHub Pull Requests 拡張 GitHub Copilot git push / git pull gh CLI SSH クライアント
② Storage API
保管を引き受けるレイヤ（誰が保存のお作法を決めているか）
クライアントごとに使うストレージ API が違う。ここが「VSCode 側か、OS 側か」の境目。
VSCode SecretStorage API Git Credential Manager (GCM) gh config file ~/.ssh/ (ファイル)
③ OS Storage
実際の物理保存先（暗号化された金庫）
VSCode SecretStorage も GCM も、最終的にはこの OS 側の金庫に書き込む。
🪟 Windows Credential Manager (DPAPI) 🍎 macOS Keychain 🐧 libsecret / gnome-keyring
重要な視点: 「VSCode 側に保存されている」と言っても、物理的には OS 側に書かれます。 厳密に分けるなら「どの API 経由で読み書きされるか」が VSCode 側 / Git 側 の違いで、 「最終的にどこに記録されるか」は基本的に全部 OS キーチェーンです。

VSCode 側 vs OS 側の役割分担

① VSCode 側（アプリ内経路）
VSCode SecretStorage
VSCode の拡張機能が使う統一的なシークレット保管 API

ここに入るもの

  • GitHub 拡張の OAuth アクセストークン（Accounts パネルでサインインしたときの鍵）
  • GitHub Pull Requests 拡張のトークン
  • GitHub Copilot の認証情報
  • Settings Sync のサインイン情報

特徴

  • VSCode の「Accounts」ペインで一元管理できる
  • 拡張機能から読むには VSCode API 経由のみ（普通のファイル読み取りでは取れない）
  • 裏側で OS キーチェーンに書き込む（詳細は下）
Backend OS キーチェーン
② OS 側（git CLI 経路）
Git Credential Manager
git.exe が credential.helper 経由で呼び出す認証ヘルパー

ここに入るもの

  • git push / git clone / git pull で HTTPS 認証したユーザー名とトークン
  • 複数リモート分（GitHub / GitLab / Bitbucket など）

特徴

  • VSCode のサインインとは完全に別
  • ターミナルから git push すると、VSCode でのサインイン状態にかかわらずこちらが使われる
  • OS キーチェーンに 「git:https://github.com」という形で直接保存
Backend OS キーチェーン（同上）
よくある誤解: 「VSCode で GitHub からサインアウトしたから、もう push できないはず」は間違い。 VSCode 側のサインアウトは VSCode SecretStorage のエントリだけを消します。 git push で使うトークンは Git Credential Manager の別エントリなので、そのまま残って機能し続けます。

リクエストが通る経路（誰がどこを見に行くか）

ケース A: VSCode の「Source Control」ビューから push したとき

1 VSCode が内部の git バイナリを subprocess として起動 VSCode
2 git が credential.helper=manager を参照して Git Credential Manager を呼び出す Git CLI
3 GCM が OS キーチェーンから git:https://github.com エントリを取得 OS Keychain
4 取得したトークンで GitHub に HTTPS 認証 GitHub

※ 途中で VSCode SecretStorage は経由していません。つまり VSCode のサインイン状態とは無関係。

ケース B: GitHub Pull Requests 拡張で PR を一覧表示したとき

1 拡張機能が vscode.authentication.getSession('github', ...) を呼ぶ VSCode Ext
2 VSCode 本体が SecretStorage から GitHub 拡張のトークンを取り出す VSCode
3 SecretStorage が OS キーチェーンの自分用エントリを復号 OS Keychain
4 取得したトークンで GitHub REST/GraphQL API を叩く GitHub

※ こちらは Git Credential Manager を一切通らない。VSCode 側で完結した独立経路。

OS 別の具体的な保存場所

Windows 資格情報マネージャー（Credential Manager）

Windows では、VSCode の認証情報も git CLI の認証情報も、最終的に同じ「資格情報マネージャー」に入ります。 ただしエントリ名（インターネットまたはネットワークのアドレス）で見分けられる作りです。

開き方: Win + R → control /name Microsoft.CredentialManager、または「Windows 資格情報」タブ。

git push 用のトークン Git CLI 系
Git Credential Manager が保存。git push / git clone https://... で使われる。
Address: git:https://github.com User: <GitHub ユーザー名> Password: <Personal Access Token または OAuth token>
VSCode GitHub 拡張のトークン VSCode 系
VSCode SecretStorage が保存。Accounts ペインのサインイン情報。
Address: vscode-github.login / vscodegithub / 類似のプレフィックス User: Code / code-oss Password: <OAuth トークン（暗号化済み）>
gh CLI のトークン（場合による） gh CLI 系
gh auth login で secure storage を選んだ場合にここへ。選ばなかった場合は下のファイルに平文で入る。
%APPDATA%\GitHub CLI\hosts.yml ← ファイル派（デフォルト） gh:github.com ← 資格情報マネージャー派

物理ファイルの場所

# 資格情報マネージャーの実体（DPAPI で暗号化）
%LOCALAPPDATA%\Microsoft\Credentials\

# Git グローバル設定（どのヘルパーを使うかだけ。トークン本体は入っていない）
%USERPROFILE%\.gitconfig

# gh CLI 設定（secure storage を使わない場合、ここに平文トークン）
%APPDATA%\GitHub CLI\hosts.yml

# SSH 鍵
%USERPROFILE%\.ssh\id_ed25519
%USERPROFILE%\.ssh\id_ed25519.pub

コマンドラインから中身を確認

# 資格情報マネージャーのエントリを一覧
cmdkey /list

# git:https://github.com だけを見る
cmdkey /list:git:https://github.com

# git が今どの helper を使っているか
git config --get credential.helper
DPAPI とは: Windows Data Protection API。 ログインユーザーのパスワードを鍵として AES 暗号化するため、別ユーザーや別マシンからは復号できません。 ゆえに「資格情報マネージャーのファイルをコピーすれば盗める」ということはない作り。

macOS キーチェーン（Keychain）

macOS でも VSCode と git CLI は同じ「login キーチェーン」に保存します。 「キーチェーンアクセス.app」を開けば GUI で確認・削除できます。

開き方: ⌘ + Space → 「キーチェーンアクセス」/ Keychain Access。

git push 用のトークン Git CLI 系
Git Credential Manager（または osxkeychain ヘルパー）が保存。
Kind: Internet password Where: https://github.com Name : git:https://github.com Account: <GitHub ユーザー名>
VSCode GitHub 拡張のトークン VSCode 系
VSCode SecretStorage が保存。
Kind: application password Name : Code - vscode.github / Code Safe Storage Account: Code / vscode-github
gh CLI のトークン gh CLI 系
gh auth login でキーチェーン格納を選ぶとここに。ファイル派ならプレーンな YAML に。
~/.config/gh/hosts.yml ← ファイル派（デフォルト） Keychain "gh:github.com" ← キーチェーン派

物理ファイルの場所

# ログインキーチェーンの実体（Apple の独自暗号化）
~/Library/Keychains/login.keychain-db

# Git グローバル設定
~/.gitconfig

# gh CLI 設定
~/.config/gh/hosts.yml

# SSH 鍵
~/.ssh/id_ed25519
~/.ssh/id_ed25519.pub

コマンドラインから中身を確認

# GitHub 用のインターネットパスワードを検索
security find-internet-password -s github.com

# git が今どの helper を使っているか
git config --get credential.helper
# 例: osxkeychain / manager-core / !/usr/local/bin/gcm など
キーチェーンは「ログインパスワードでロック解除」される金庫: ログイン中は暗黙に unlock されているので git push はスムーズ。 スリープ復帰直後などに「キーチェーンのロックを解除」ダイアログが出るのはこのため。

Linux — libsecret / gnome-keyring / KWallet

Linux は標準の資格情報ストアが存在せず、ディストリやデスクトップ環境によって変わります。 VSCode は libsecret（GNOME: gnome-keyring / KDE: kwallet）経由で保存しようとしますが、 ヘッドレスサーバーなど keyring が動いていない環境では暗号化なしのファイルにフォールバックすることも。

物理ファイルの場所（代表例）

# gnome-keyring の DB（暗号化済み）
~/.local/share/keyrings/login.keyring

# VSCode の設定
~/.config/Code/

# Git グローバル設定
~/.gitconfig

# gh CLI 設定
~/.config/gh/hosts.yml

# SSH 鍵
~/.ssh/id_ed25519
ヘッドレス/SSH のハマりどころ: keyring デーモンが動いていないリモートサーバーで VSCode Server を使うと、 トークンが 暗号化されないファイルに置かれる可能性があります。 root ユーザーや共用サーバーで SecretStorage を使うのは避けるか、専用の credential helper を導入してください。

確認・サインアウト・完全削除のやり方

VSCode 側のサインアウト

サイドバー左下の人型アイコン（Accounts）→ GitHub の「…」→ Sign Out。

これで消えるのは VSCode SecretStorage の GitHub エントリのみ。Git CLI 側のトークンは残る。

Git CLI 側（push 認証）のリセット

# 保存されているトークンを削除（一度だけ再入力が要求される）
git credential reject <<< $'protocol=https\nhost=github.com'

# Windows: 資格情報マネージャーから直接削除
cmdkey /delete:git:https://github.com

# macOS: セキュリティコマンドで削除
security delete-internet-password -s github.com

gh CLI のログアウト

gh auth logout --hostname github.com
完全にきれいにしたい場合: VSCode サインアウト / git credential 削除 / gh auth logout / ~/.ssh の整理 — この 4 つを別個にやる必要があると覚えておけば OK。

よくあるハマりポイント

Q. VSCode でサインアウトしたのに、なぜか push できてしまう
A. git push は Git Credential Manager 経由。VSCode のサインインとは別ストア。 cmdkey /delete:git:https://github.com（Win）や security delete-internet-password -s github.com（Mac）で git 側も消す必要があります。
Q. git push しようとしたら VSCode がブラウザを開いてサインインを求めてくる
A. Git Credential Manager の新しいバージョンは、OS キーチェーンが空のとき OAuth フローでトークンを取得し、それを保存します。 意図して OAuth を使いたくない場合は、Personal Access Token で一度 push するか、credential.credentialStore の設定を確認。
Q. 別アカウントを使い分けたい
A. credential.useHttpPath=true を設定してリポジトリ URL 単位で認証情報を分けるか、 リモートごとに SSH 鍵を分ける方法が安定。VSCode 側のアカウント切替（Accounts → Switch to a different account）とは別に、git 側の切替も必要。
Q. PC を譲渡・廃棄する前にやるべきこと
A. 「ファイルを消した」だけではダメ。少なくとも以下を明示的にクリアすること:
  • Windows 資格情報マネージャーの git:* / vscode-* 系エントリ削除
  • macOS キーチェーンの GitHub 関連エントリ削除
  • ~/.gitconfig / ~/.ssh/ / ~/.config/gh/ を削除
  • GitHub 側で Personal Access Token を revoke（ブラウザから Settings → Developer settings）

安全運用のチェックリスト

  • gh CLI は secure storage を選ぶ — gh auth login 時に「Store your credential in secure storage?」で Yes を選べば OS キーチェーンに入る。
  • SSH 鍵にはパスフレーズを設定 — ~/.ssh/ のファイルは OS キーチェーンに入らないので、鍵そのものを暗号化しておく。
  • Personal Access Token は fine-grained で最小権限 — repo 全権ではなく必要なスコープだけに絞る。
  • 共用 PC では VSCode SecretStorage に依存しない — 同じ OS ユーザーを共有すると拡張機能は当然読める。
  • トークン期限を設定 — 無期限トークンは、漏れたときに長期間悪用される。90 日などの期限を。
  • 定期的に GitHub の Settings → Security log を確認 — 不審な IP からのアクセスがないか。

まとめ

「GitHub の認証情報は VSCode 側か、OS 側か」への答えを 1 枚に。

論点 答え
物理的な保存先（実ファイル） 基本は OS 側（Windows 資格情報マネージャー / macOS キーチェーン）。VSCode にも git CLI にも専用ファイルはない。
論理的な保存先（どの API が管理） 用途で分かれる。拡張機能系は VSCode SecretStorage、git push は Git Credential Manager、gh は独自。
VSCode からサインアウトすれば全部消える？ No。VSCode 管轄の SecretStorage エントリだけ。git push 用は別に消す必要あり。
Windows と Mac で場所が違う？ 物理ファイルは違うが構造は同じ。資格情報マネージャー ⇄ login.keychain の対応。
平文で保存される可能性は？ 基本なし。ただし gh CLI の hosts.yml はデフォルトで平文、Linux の keyring なし環境も要注意。
