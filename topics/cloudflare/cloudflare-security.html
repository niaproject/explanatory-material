Cloudflare セキュリティ設定の決定版

セキュリティ 3 本柱の全体像▾

Cloudflare のセキュリティ設定は数十項目あって最初は何から触れば良いか迷いますが、 まず以下の 3 つを順番に入れるだけで 「最低ライン」 はクリアできます。

① 暗号化最大化
🔐 SSL/TLS を Full (strict) に
ブラウザ ↔ Cloudflare ↔ オリジンの全区間を TLS で結び、 オリジン証明書も検証する最も厳格なモードへ引き上げる。
② HTTPS 自動化
🔁 Always HTTPS + HSTS
全 HTTP アクセスを 301 で HTTPS に強制リダイレクトし、 さらに HSTS でブラウザに「次回から最初から HTTPS」と覚えさせる。
③ ボット対策
🤖 Bot Fight Mode + WAF
無料の Bot Fight Mode を ON、Browser Integrity Check と マネージド WAF ルール、Rate Limiting を併用してスクレイパー・スキャナーを排除。

① 暗号化を最大化する（SSL/TLS）▾

Cloudflare ↔ オリジンサーバ間の暗号化レベルを決める設定で、 ここを誤ると 「鍵マークは付いているのに実は素の HTTP で通信していた」 という事故になります。

SSL/TLS 暗号化モードの違い

モード ブラウザ ↔ CF CF ↔ オリジン 証明書検証 備考
Off HTTP HTTP — 禁忌。常時無効化。
Flexible HTTPS HTTP — 後半が平文。リダイレクトループの温床。使ってはいけない。
Full HTTPS HTTPS 無し 自己署名証明書を許容。中間者攻撃に弱い。
Full (strict) ✅ HTTPS HTTPS あり 本番の正解。オリジンに正規証明書 or Cloudflare Origin CA を入れる。
Strict (SSL-Only Origin Pull) HTTPS mTLS あり Authenticated Origin Pulls。Cloudflare 経由以外を遮断。
⚠ Flexible SSL は絶対に避ける：「サイトは HTTPS 化したように見えて、 実は CF→オリジン間が裸の HTTP」という最悪のパターンになります。WordPress 等で リダイレクトループ（ERR_TOO_MANY_REDIRECTS）が出る典型原因でもあります。
1 SSL/TLS 暗号化モードを Full (strict) に切り替え
ダッシュボード › ドメイン選択 › SSL/TLS › Overview
  • 「Custom SSL/TLS」を Full (strict) に変更
  • オリジンに有効な証明書が無い場合は、SSL/TLS → Origin Server から Create Certificate で Cloudflare Origin CA 証明書（最大 15 年）を発行してオリジンへ設置
2 最低 TLS バージョンを 1.2 以上に
SSL/TLS › Edge Certificates › Minimum TLS Version
✗ 古い
TLS 1.0 / 1.1
PCI DSS 非準拠。BEAST/POODLE などの既知攻撃あり。
✓ 推奨
TLS 1.2（最低） / TLS 1.3（推奨）
古い IE / Android 4.x を切れるなら 1.3 が理想。
3 TLS 1.3 と Opportunistic Encryption を ON
SSL/TLS › Edge Certificates
  • TLS 1.3: ON（高速・安全な最新プロトコル）
  • Opportunistic Encryption: ON（HTTP 要求も裏で HTTPS で取りに行く）
  • Automatic HTTPS Rewrites: ON（HTML 内の http:// リンクを自動で https:// に書換）
💡 Authenticated Origin Pulls（mTLS）まで入れると完璧： SSL/TLS → Origin Server → Authenticated Origin Pulls を ON にし、 オリジン側で cloudflare.crt を信頼すると Cloudflare 経由以外の HTTPS リクエストをオリジンが拒否できます（IP バイパス防止）。

② HTTPS を自動化する（強制リダイレクト & HSTS）▾

オリジンとの間が暗号化されても、ユーザが http:// で来てしまえば 最初の 1 リクエストだけは平文。これを防ぐのが 常時 HTTPS + HSTS の組み合わせです。

1 Always Use HTTPS（HTTP→HTTPS 強制リダイレクト）
SSL/TLS › Edge Certificates › Always Use HTTPS

ON にすると、Cloudflare が http:// へのリクエストを 自動で 301 → https:// にリダイレクトします。 アプリ側のリダイレクト実装が不要になり、設定漏れによる平文アクセスを完全に塞げます。

2 HSTS（Strict-Transport-Security）の有効化
SSL/TLS › Edge Certificates › HTTP Strict Transport Security (HSTS)

HSTS はブラウザに「このドメインは 今後一切 HTTP で来るな」と覚えさせるヘッダです。

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
  • Max Age: 6 months（15552000） から開始 → 問題なければ 12 months に延長
  • Apply HSTS to subdomains (includeSubDomains): 全サブドメインを HTTPS 化済みなら ON
  • Preload: hstspreload.org に登録すると、 ブラウザのプリロードリストに焼き込まれ初回アクセスから HTTPS 強制になる
  • No-Sniff Header: ON 推奨
⚠ HSTS の解除は事実上できない：max-age を長くすると、 設定ミスで HTTPS が止まったときユーザのブラウザに「最大 N 秒間 HTTP に戻れない」記憶が残ります。 最初は短い max-age（例: 300 秒）で挙動確認 → 段階的に伸ばすのが安全です。
3 Universal SSL（無料の Edge 証明書）が発行済みか確認
SSL/TLS › Edge Certificates › Universal SSL

ドメインを Cloudflare の DNS に向けた瞬間に自動発行・自動更新されます。 ステータスが Active でない場合は、DNS の CAA レコード設定や ネームサーバ未切替が原因。

③ ボット対策を有効化する▾

Cloudflare のボット対策は、無料でもかなり強力です。 新ダッシュボードでは Security → Settings ページに各種トグルが集約され、 Web application exploits / DDoS attacks / Bot traffic / API abuse / Client-side abuse の 5 カテゴリに整理されています。

⚠ Security Level（Low / Medium / High）は事実上の非推奨： 旧ダッシュボードにあった Security Level の判定根拠だった threat score（0–100）は現在常に 0 しか返さなくなっており、 Cloudflare 公式が 「threat score ベースのルールは作るな」 と明言しています。新ダッシュボードでは Security Level の UI 自体が見当たらないことが多く、 残っている場合も 実質的には "Under Attack Mode の ON/OFF スイッチ" としてしか機能しません。 本ページでは Security Level に依存せず、Bot Fight Mode + WAF Managed Rules + Rate Limiting の組み合わせで防御する手順に統一しています。
1 Bot Fight Mode を ON
Security › Settings › Bot traffic › Bot Fight Mode

Cloudflare が「明らかにボット」と判定したリクエストに対し 自動で計算チャレンジ（Proof-of-Work）を返します。Free プランで利用可能。 Pro 以上では Super Bot Fight Mode が選べ、 Likely Automated / Verified Bot を細かく制御できます。

ℹ Verified Bots は許可される：Googlebot / Bingbot / Slackbot のように Cloudflare が検証済みのクローラはブロック対象外なので、SEO に影響しません。
ℹ Security → Settings ページの構成： 新ダッシュボードでは Security → Settings ページに、 Browser Integrity Check / Challenge Passage / Replace Insecure JS Libraries / Email Address Obfuscation / Hotlink Protection などのトグルがフラットに並びます。 上部の カテゴリフィルタ（Web application exploits / DDoS attacks / Bot traffic / API abuse / Client-side abuse） で絞り込めますが、フィルタを外せば全項目が一覧で見えます。各設定はトグル式で、 横に鉛筆アイコンがある項目はクリックで詳細値を変更できます。
直リンク: https://dash.cloudflare.com/?to=/:account/:zone/security/settings
2 Browser Integrity Check を ON
Security › Settings › Browser Integrity Check

User-Agent や HTTP ヘッダに既知の異常パターン（スパムボットでよく見られる 壊れた User-Agent、欠落必須ヘッダなど）があるリクエストを自動的に弾く機能です。 誤検知が起きにくく副作用がほぼ無いので、ON 固定で問題ありません。

3 Challenge Passage の有効期間を確認
Security › Settings › Challenge Passage

チャレンジ（CAPTCHA / JS チャレンジ）に一度通過したクライアントを、 どのくらいの間「人間扱い」して再チャレンジを免除するかを決める設定です。 デフォルトは 30 分。Cloudflare 推奨レンジは 15〜45 分で、 短すぎるとログインユーザに何度もチャレンジが出てしまい、 長すぎると共有 IP（オフィス NAT・モバイル回線）から ボットが免除を引き継いでしまうリスクがあります。

4 Replace Insecure JavaScript Libraries を ON
Security › Settings › Replace Insecure JavaScript Libraries

ページ内で参照している JS ライブラリのうち、 既知の脆弱性が報告されている古いバージョン（例: 古い jQuery）を、 Cloudflare の Edge で cdnjs ホストの安全版に自動的に書き換えて配信する機能です。 サードパーティテーマや古い WordPress プラグインに引きずられて 古い JS を読み込んでしまうサイトには特に有効。ON 推奨。

5 緊急時のみ "Under Attack Mode" を ON
ゾーン Overview ページ › 右側 Quick Actions サイドバー › Under Attack Mode

旧 Security Level の "I'm Under Attack!" に相当する機能で、 全アクセスに JS チャレンジを挟みます。 新ダッシュボードでは Security メニュー配下ではなく、ゾーン Overview ページ右側の Quick Actions に独立して配置されました。サイトが重くなり API クライアントも弾かれるため、 DDoS を実際に受けているときだけ一時的に ON にし、収束後すぐ OFF に戻すのが原則。 平常時は触らないこと。

6 フォーム/ログインに Turnstile を入れる
アカウントホーム左サイドバー › Turnstile › Add widget

reCAPTCHA の代替となる Cloudflare 公式の無料 CAPTCHA。 アカウント単位で Add widget から sitekey / secret を発行し、サイトに <div class="cf-turnstile" data-sitekey="..."></div> を埋め込み、 サーバ側で POST されたトークンを検証 API で検証します。 ユーザ操作不要（不可視チャレンジ）でフォーム自動投稿を弾けます。 

# Turnstile の検証（サーバサイド）
curl -X POST "https://challenges.cloudflare.com/turnstile/v0/siteverify" \
  -d "secret=$TURNSTILE_SECRET" \
  -d "response=$TOKEN_FROM_FORM"

直リンク: https://dash.cloudflare.com/?to=/:account/turnstile

7 レート制限ルールでログイン総当たりを止める
Security › Security rules › Create rule › Rate limiting rules

新ダッシュボードでは Rate Limiting / Custom Rules はかつての WAF ページから 「Security rules」ページに移動・統合されました。Free プランでも 1 ルール まで Rate Limiting が組めます。 /login や /wp-login.php のような高リスクパスに 「同一 IP から 1 分 10 リクエスト超でブロック」を入れるのが定番。 

# 例: /login へ 1分 10 回超えたらブロック
URI Path equals "/login"
Action: Block
Period: 1 minute
Requests: 10

直リンク: https://dash.cloudflare.com/?to=/:account/:zone/security/security-rules

+ 国別アクセス制限（特定国のみ許可 / ブロック）▾

「日本からのアクセスのみ許可したい」「中国・ロシアからは丸ごと弾きたい」といった 地理的アクセス制御（Geo-blocking） は、Cloudflare の Custom Rules（旧 Firewall Rules / WAF Custom Rules）で実現します。 Free プランでも 5 ルールまで無料で組めます。

⚠ Cloudflare には「許可リストモード」が無い： 「許可した国以外を全部ブロック」というデフォルト Deny の挙動はネイティブ提供されていません。 そのため 「許可国 *以外* を Block」という否定形ルール として書くのが定石です。 AWS の Security Group 的な「Allow を最後に置けばデフォルト Deny」とは設計思想が違うことに注意。
1 Custom Rules ページを開く
Security › Security rules › Create rule › Custom rules

旧ダッシュボードでは「Firewall Rules」「WAF › Tools」と呼ばれていた領域が、 新ダッシュボードでは Security rules ページに Rate Limiting と一緒に統合されています。

直リンク: https://dash.cloudflare.com/?to=/:account/:zone/security/security-rules

2 「特定国のみ許可」する式（ホワイトリスト方式）

許可したい国を not ... in {...} で囲んで Block するのが基本形。 これで「日本以外は全部ブロック = 日本のみ許可」が成立します。

パターン A: 日本のみ許可

# 式 (Expression)
(ip.geoip.country ne "JP")

# Action
Block

パターン B: 日本＋米国のみ許可（実務での推奨ライン）

# 式 (Expression)
(not ip.geoip.country in {"JP" "US"})

# Action
Block
💡 なぜ「JP のみ」より「JP + US」が現実的か： Googlebot / Bingbot はほぼ 米国 IP 帯から来ます。JP のみにすると SEO クローラが 全滅して数日でインデックスから落ちます。US を足すだけで Googlebot / Bingbot / Slack・X・Discord の OGP プレビュー / 多くの SaaS Webhook（Stripe / SendGrid 等） が自動的に通り、それでいて中国・ロシア・東欧由来の攻撃の大半は遮断できます。

パターン C: 管理画面 /admin だけ国制限（推奨）

# 公開ページは全世界、管理画面だけ JP に絞るパターン
(http.request.uri.path contains "/admin"
  and not ip.geoip.country in {"JP"})

# Action
Block
💡 サイト全体より「特定パスだけ国制限」の方が安全： 公開コンテンツを国でブロックすると SEO クローラ・海外正規ユーザ・出張中の自社ユーザを 巻き込みます。/wp-admin /admin /api/internal など 本来海外から触られる必要のないパスだけに絞ると副作用が出ません。
3 「特定国だけブロック」する式（ブラックリスト方式）

逆に「日本向けサービスだけど、攻撃の多い国だけ弾きたい」というケースは 肯定形の in {...} で Block します。 

# 例: CN / RU / KP からのアクセスをブロック
(ip.geoip.country in {"CN" "RU" "KP"})

# Action
Block

国コードは ISO 3166-1 alpha-2（JP / US / GB / DE / CN / KR / TW …）。 GUI のフィールドエディタなら国名選択 UI から選べるのでコードを覚える必要はありません。

4 いきなり Block ではなく Managed Challenge から始める
✗ いきなり
Action: Block
VPN / Cloudflare WARP で国判定がブレた正規ユーザを巻き込みやすい。
✓ 段階導入
Action: Managed Challenge
1〜2 日 Security Events ログを観察 → 誤検知が無いことを確認してから Block へ昇格。

ログは Security › Events から国別の発生件数が見られるので、 正規ユーザ層の国分布を把握してからルールを締めるのが安全です。

5 「JP のみ」の場合は Verified Bots を除外する（JP+US ならほぼ不要）

Googlebot / Bingbot は米国 IP 帯から来るため、 パターン B（JP + US 許可）を採用していれば 地理条件側で自動的に通るので Verified Bots 除外は基本不要です。一方パターン A（JP のみ）にすると Googlebot まで 弾いて SEO が壊滅するため、cf.client.bot 除外が必須になります。

JP のみ許可の場合（cf.client.bot 除外が必須）

# 日本以外を Block、ただし Verified Bots は許可
(not ip.geoip.country in {"JP"}
  and not cf.client.bot)

# Action
Block
⚠ cf.client.bot 除外は YandexBot / Baiduspider まで通してしまう： Cloudflare の Verified Bots には Googlebot / Bingbot だけでなく YandexBot（露）/ Baiduspider（中）も含まれます。日本ターゲットのサイトで これらを通す価値はほぼ無く、むしろクロール負荷が高い厄介者です。 JP + US 許可なら Googlebot / Bingbot は地理条件で既に通るので、 「cf.client.bot 除外を入れない方が Yandex / Baidu を弾けて合理的」 になります。なお DuckDuckGo は実体が Bing インデックスのリブランドなので、 DuckDuckBot を弾いても DuckDuckGo の検索順位には影響しません。
ℹ より精密に bot を選別したい場合： cf.verified_bot_category で bot をカテゴリ単位で許可できます。例: cf.verified_bot_category in {"Search Engine Crawler" "Monitoring & Analytics"} とすれば検索エンジンと監視系だけ通せます。
ℹ Tor / 匿名プロキシは特殊コード T1： Cloudflare は Tor 出口ノード経由を国コード T1 として返します。 完全に塞ぎたいなら別ルールで (ip.geoip.country eq "T1") を Block / Managed Challenge にすると分かりやすく管理できます。
⚠ 国判定は 100% 正確ではない： GeoIP DB はモバイルキャリアの CGNAT、衛星回線、新規割当 IP で誤判定が起きます。 また Cloudflare WARP / iCloud Private Relay 経由のユーザは出口リレーの国で判定されるため、 国内ユーザでも US / GB 判定になることがあります。「国制限 = 完全な防御」と思わず、 あくまで攻撃面を減らす一手段として使ってください。

+α 合わせて設定すべきもの▾

上記 3 本柱に加え、攻撃面を一段階下げるために以下も入れておくと実質 「やれることは無料枠で全部やった」状態になります。

🛡 WAF マネージドルール

OWASP Top 10 などの既知攻撃パターンを自動でブロック。Free でも Cloudflare Free Managed Ruleset が使える。

Security › WAF › Managed rules

🚫 カスタム WAF ルール

国別ブロック / 特定 User-Agent ブロック / wp-admin を自宅 IP のみ許可、 などをノーコードで追加できる。新ダッシュボードでは Rate Limiting と同じ Security rules ページに統合された。

Security › Security rules › Create rule › Custom rules

🔑 DNSSEC を ON

DNS レスポンスの改ざん（キャッシュポイズニング）を署名で検証。 ボタン 1 つで有効化できるが、レジストラ側に DS レコード登録が必要。

DNS › Settings › DNSSEC

📧 Email Address Obfuscation

HTML 中の mailto: をスクレイパー対策で自動難読化。 メールアドレス収集系ボットの収穫率を激減させる。デフォルト ON。

Security › Settings › Email Address Obfuscation

🔗 Hotlink Protection

画像直リンク（外部サイトに <img> 埋込）を遮断。 画像転送量の盗用と帯域コストを防ぐ。

Security › Settings › Hotlink Protection

🌐 .onion ルーティング（任意）

Tor ユーザ向けに .onion アドレスを自動発行・対応。 プライバシー重視メディアでは有用。一般サイトでは不要。

Network › Onion Routing

🚷 オリジン IP の隠蔽

DNS A レコードは必ず オレンジ雲（Proxied）に。 グレー雲（DNS Only）だとオリジン IP が漏れて Cloudflare 迂回攻撃の踏み台になる。

DNS › Records › 各レコードの Proxy 状態

🔒 アカウントの 2FA

Cloudflare アカウント自体を奪われると DNS / 証明書 / WAF を全部書き換えられる。 TOTP もしくは WebAuthn の 2 要素認証を必ず ON。

右上アイコン › My Profile › Authentication

📜 Audit Logs を定期確認

誰がいつどの設定を変えたかが残る。設定改ざん検知 / 共同管理者の事故防止に。

Manage Account › Audit Log

🔄 Page Shield（Pro+）

サイトに読み込まれる JavaScript を監視し、 サプライチェーン経由のスキマー（Magecart 等）を検知。

Security › Page Shield

最終チェックリスト▾

新規ドメインを Cloudflare に乗せた直後、上から順に潰していけば抜けが出ません。

  • SSL/TLS モード を Full (strict) に設定した
  • Minimum TLS Version を 1.2 以上に上げた
  • TLS 1.3 / Opportunistic Encryption / Automatic HTTPS Rewrites を ON にした
  • Always Use HTTPS を ON にして HTTP→HTTPS 強制リダイレクトを有効にした
  • HSTS を有効化（最初は短い max-age → 段階的に延長）
  • Universal SSL が Active になっていることを確認した
  • Bot Fight Mode を ON にした（Security Level は threat score 廃止により実質非推奨なので触らない）
  • Browser Integrity Check / Privacy Pass を ON にした
  • ログイン系パスに Rate Limiting ルールを 1 つ入れた
  • WAF Managed Rules（Free） を有効化した
  • 必要に応じて 国別アクセス制限（Custom Rules）を入れた（管理画面パスのみ等）
  • DNS レコードがすべて オレンジ雲（Proxied） になっている
  • DNSSEC を有効化し、レジストラ側に DS レコードを登録した
  • Cloudflare アカウントに 2FA をかけた
