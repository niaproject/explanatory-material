conntrack -L / ovs-ofctl dump-flows / ovn-trace

3 つの主要デバッグコマンドが「どのレイヤを覗くのか」を整理する

このページの結論（TL;DR）

どのコマンドを、いつ使うか

conntrack -L は Linux カーネルの接続追跡テーブルを見るコマンド。「NAT / ステートフル FW が実際にどの接続を保持しているか」を知りたいときに使う。

ovs-ofctl dump-flows は OVS の OpenFlow フローテーブルを表示するコマンド。「各パケットがどのルールにマッチして、どのカウンタが増えているか」を知りたいときに使う。

ovn-trace は OVN の論理フロー上で仮想パケットをシミュレーションするコマンド。「実際に流さずに、このパケットは論理的にどう処理されるか」を知りたいときに使う。

conntrack -L

Linux kernel の接続状態を覗く
kernel nf_conntrack 実測値

ovs-ofctl dump-flows

OpenFlow フローと統計カウンタを覗く
OVS OpenFlow 実測値

ovn-trace

論理ネットの通過経路をシミュレート
OVN Logical Flow ドライラン

3 コマンドが覗いている「レイヤ」の違い

同じネットワーク問題でも、どのレイヤが怪しいかで使うコマンドが変わります。下図の通り、OVN / OVS / Linux kernel それぞれ別のデータ源を持っているため、適切な層を選ばないと「正しい答えが見えない」状況になります。

1. conntrack -L — Linux カーネルの接続追跡を覗く

目的

Linux カーネルの nf_conntrack モジュールが保持している 接続トラッキングテーブルのスナップショットを表示します。 ステートフルファイアウォール・NAT・Load Balancer の「今どの接続を覚えているか」が分かります。

パッケージ: conntrack-tools（Debian/Ubuntu: apt install conntrack、RHEL 系: dnf install conntrack-tools）。root 権限が必要です。

基本形と出力フォーマット

# 全エントリを表示
sudo conntrack -L

# 代表的な出力行（TCP で 192.168.1.10:52430 → 10.0.0.5:443 に張られた接続）
tcp      6 431999 ESTABLISHED src=192.168.1.10 dst=10.0.0.5 sport=52430 dport=443 \
                              src=10.0.0.5 dst=192.168.1.10 sport=443 dport=52430 \
                              [ASSURED] mark=0 use=1

各フィールドの意味

tcp ← L4 プロトコル名
6 ← プロトコル番号（/etc/protocols と一致）
431999 ← このエントリの残り TTL 秒（既定 ESTABLISHED = 5 日 = 432000）
ESTABLISHED ← TCP ステート（NEW/SYN_SENT/ESTABLISHED/FIN_WAIT/TIME_WAIT …）
src=... ← ❶ 往路の 5 タプル（original direction）
src=... ← ❷ 復路の 5 タプル（reply direction）
[ASSURED] ← 両方向観測済みフラグ（テーブル満杯時も守られやすい）
mark=0 ← conntrack mark（iptables CONNMARK / OVS ct_mark で付与）
use=1 ← このエントリを参照している kernel 構造体の数
2 行ペアで 1 エントリ。 1 つの接続につき「往路」と「復路」の 5 タプルが並びます。NAT されている場合は、復路の IP / ポートが 書き換え後の値になるため、一目で NAT の方向と変換先が判別できます。

よく使うフィルタ

# プロトコル指定
sudo conntrack -L -p tcp
sudo conntrack -L -p udp

# 特定の状態だけ
sudo conntrack -L -p tcp --state ESTABLISHED
sudo conntrack -L -p tcp --state SYN_SENT      # ハーフオープン

# 宛先 IP / 送信元 IP で絞る
sudo conntrack -L -d 10.0.0.5
sudo conntrack -L -s 192.168.1.10 -p tcp --dport 443

# NAT 関連だけ（NAT 変換が起きているエントリのみ）
sudo conntrack -L --src-nat
sudo conntrack -L --dst-nat

# 件数だけ欲しい（table full 調査）
sudo conntrack -C
cat /proc/sys/net/netfilter/nf_conntrack_count
cat /proc/sys/net/netfilter/nf_conntrack_max

# リアルタイム監視（新規/更新イベントを流し続ける）
sudo conntrack -E

実例 1: Pod → 外部 DB への通信が「行けているのに戻ってこない」

Kubernetes ノード上で Pod (10.244.3.7) が 外部 PostgreSQL (10.0.50.20:5432) に繋がらない。この時に真っ先に見るべきは「そもそも kube-proxy / OVN NAT が効いているか」です。

# Pod が張ろうとした TCP 接続を conntrack 側で観測
sudo conntrack -L -p tcp --dport 5432 | grep 10.244.3.7

── 期待する出力 ──
tcp 6 118 SYN_SENT src=10.244.3.7 dst=10.0.50.20 sport=41100 dport=5432 \
                   [UNREPLIED] src=10.0.50.20 dst=203.0.113.7 sport=5432 dport=41100 mark=0 use=1
                                                        ^^^^^^^^^^^^
                                                        NAT 後の送信元（ノードの外部 IP）
  • SYN_SENT [UNREPLIED] → 外向きに出してはいるが、応答が戻ってない＝経路 or ファイアウォールが怪しい
  • 復路の src= が Pod IP のままだったら SNAT が効いていない → kube-proxy / OVN の LB 設定を疑う

実例 2: 「突然パケットが落ちる」— table full を疑う

dmesg | grep -i conntrack
[12345.6789] nf_conntrack: nf_conntrack: table full, dropping packet

cat /proc/sys/net/netfilter/nf_conntrack_count
262143

cat /proc/sys/net/netfilter/nf_conntrack_max
262144

# 対処: 上限を引き上げる（永続化は /etc/sysctl.d/ に書く）
sudo sysctl -w net.netfilter.nf_conntrack_max=524288

# どの状態がテーブルを食っているか集計
sudo conntrack -L 2>/dev/null | awk '{print $3}' | sort | uniq -c | sort -rn
  181203 TIME_WAIT     ← これが多すぎならタイムアウト短縮を検討
   72110 ESTABLISHED
   8830 SYN_SENT
TIME_WAIT が支配的な場合は net.netfilter.nf_conntrack_tcp_timeout_time_wait（既定 120 秒）を短縮するか、トラフィック次第では -j NOTRACK / notrack ターゲットで一部を除外します。

実例 3: OVS で commit された接続を確認する

OVS の ct(commit, zone=5) アクションが実行されると、nf_conntrack に zone=5 で登録されます。ゾーン指定のフィルタで、OVS が書いたエントリだけを抜き出せます。

sudo conntrack -L --zone 5 -p tcp
tcp 6 431999 ESTABLISHED src=10.0.0.2 dst=10.0.0.3 sport=40000 dport=80 \
                          src=10.0.0.3 dst=10.0.0.2 sport=80 dport=40000 \
                          [ASSURED] mark=0 zone=5 use=1

2. ovs-ofctl dump-flows — OpenFlow フローと統計カウンタを覗く

目的

指定したブリッジの OpenFlow テーブルに登録されている全フロールール（match と action のペア）と、 各ルールが何回マッチし、何バイト処理したかのカウンタを表示します。 パケットが意図したフローに届いていない／別のフローに食われている、という問題の一次切り分けに使います。

基本形と出力フォーマット

# ブリッジ br-int の全フローを表示
sudo ovs-ofctl dump-flows br-int

# OVN 環境では OpenFlow 1.3 以上を指定（OVN は 1.3+ 前提）
sudo ovs-ofctl -O OpenFlow13 dump-flows br-int

── 代表的な出力行 ──
cookie=0x0, duration=1234.5s, table=0, n_packets=42, n_bytes=3528, priority=100,\
  tcp,in_port=1,nw_dst=10.0.0.5,tp_dst=443 actions=ct(table=1,zone=5),output:2

各フィールドの意味

cookie=0x0 ← コントローラ（OVN 等）が付与する識別子。OVN は Logical_Flow の UUID を埋める
duration=1234.5s ← このフローがインストールされてからの経過秒数
table=0 ← OpenFlow のテーブル番号（OVN は 0〜約 80 まで用途別に使い分け）
n_packets=42 ← このフローにマッチした累積パケット数 ★超重要
n_bytes=3528 ← 同じく累積バイト数
priority=100 ← 同テーブル内で高い値が優先（同じ match でも priority 差で勝敗が決まる）
tcp,in_port=1,nw_dst=... ← マッチ条件。1 つでも不一致があれば次のフローへ
actions=ct(...),output:2 ← マッチ時の処理。複数アクションはカンマ区切りで順次実行
n_packets が 0 のままなら、パケットはこのフローに届いていない。 デバッグで最初に見るべき列です。「フローは入っているのに通信できない」なら、priority 競合や match 不一致で別のフローが拾っている可能性を疑います。

よく使うフィルタ

# 特定テーブルだけ
sudo ovs-ofctl dump-flows br-int table=0

# 特定 in_port・特定 IP にマッチするフローだけ
sudo ovs-ofctl dump-flows br-int "in_port=5,ip,nw_dst=10.0.0.5"

# cookie（論理フロー UUID 相当）で絞る
sudo ovs-ofctl dump-flows br-int "cookie=0x3f8b/-1"  # / のあとの -1 は「マスクなし=完全一致」

# 統計だけ（match + actions は省略してサマリ表示）
sudo ovs-ofctl dump-aggregate br-int

# 指定パケットが通るルートをシミュレート（ovn-trace の OVS 版）
sudo ovs-appctl ofproto/trace br-int \
  "in_port=1,tcp,nw_src=10.0.0.2,nw_dst=10.0.0.5,tp_dst=443"

# kernel datapath の実キャッシュ（ovs-ofctl とは別レイヤ）
sudo ovs-dpctl dump-flows
ovs-ofctl dump-flows と ovs-dpctl dump-flows は別物。 前者は ユーザー空間の OpenFlow テーブル、後者は kernel datapath に落ちたキャッシュです。前者に居るのに後者に居ない場合は「upcall はあるがヒットしていない」可能性があります。

実例 1: 「フローは入っているのに通信できない」

OVN で Security Group の allow ルールを書いたのに、VM 間通信が通らないケース。

sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | grep "nw_dst=10.0.0.5"
cookie=0x11aa, duration=300s, table=44, n_packets=0, n_bytes=0,\
  priority=2002,tcp,reg15=0x3,metadata=0x5,nw_dst=10.0.0.5,tp_dst=443 \
  actions=resubmit(,45)
cookie=0x22bb, duration=320s, table=44, n_packets=18, n_bytes=1512,\
  priority=2003,ip,reg15=0x3,metadata=0x5 actions=drop
  • allow フロー（priority=2002）は n_packets=0 → パケットが到達していない
  • 一方、priority=2003 の drop にカウンタが乗っている → より高い priority が先に食っている
  • 対処: OVN の ACL 定義を見直し、allow の priority を上げる（OVN の ACL priority を 2003 以上に設定）

実例 2: conntrack commit がどのフローで起きているか

sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | grep "ct(commit"
cookie=0x33cc, duration=1200s, table=41, n_packets=532, n_bytes=38112,\
  priority=100,ct_state=+new+trk,tcp,reg15=0x3,metadata=0x5 \
  actions=ct(commit,zone=NXM_NX_REG13[0..15],exec(load:0x1->NXM_NX_CT_LABEL[0])),resubmit(,42)

このフローが nf_conntrack にエントリを書き込む箇所。conntrack -L --zone N で同じゾーンを見れば、ここから登録されたエントリが確認できます。

3. ovn-trace — 論理ネットワークをシミュレーションする

目的

OVN の 論理フロー（NB DB の設定から ovn-northd が生成した、物理情報を含まない match/action 集合）上で、 仮想パケットを流すシミュレーションを行い、どのパイプラインを通り、 どのフロー（ACL、ルーティング、LB、NAT 等）でどう処理され、最終的にどこへ出力されるか を追跡します。

特徴: 実際にはパケットを流さないため、実トラフィックを発生させずに机上で検証できる。変更前の検証・障害切り分けどちらにも使えます。

基本形と 3 つの出力モード

# 基本形: ovn-trace <datapath> <microflow>
# datapath = 論理スイッチ名 または 論理ルータ名（入り口）
sudo ovn-trace sw0 \
  'inport == "sw0-port1" && eth.src == 00:00:00:00:00:01 && eth.dst == 00:00:00:00:00:02 \
   && ip4 && ip4.src == 10.0.0.2 && ip4.dst == 10.0.0.3 \
   && ip.ttl == 64 && tcp && tcp.dst == 80'

# 出力モード（詳細度で 3 段階）
sudo ovn-trace --minimal    ...  # 最終アクション（output ポートだけ）
sudo ovn-trace --summary    ...  # パイプラインごとにまとめて表示
sudo ovn-trace --detailed   ...  # 通過した全 Logical Flow を逐次表示（デフォルト）

microflow の書き方（主要フィールド）

inport == "..." ← 入り口の論理ポート名（ovn-nbctl lsp-list で確認）
eth.src / eth.dst ← 送信元 / 宛先 MAC
ip4 / ip6 ← プロトコル指定（必須）
ip4.src / ip4.dst ← 送信元 / 宛先 IPv4
ip.ttl ← TTL（論理ルータ通過の挙動を再現するなら必須）
tcp / udp / icmp4 ← L4 指定
tcp.src / tcp.dst ← L4 ポート（ACL や LB 判定に必要）
条件は && で連結した 1 本の式として渡します。シングルクォートで全体を囲んでシェル展開を防ぐのが鉄則です。

出力の読み方（--detailed）

sudo ovn-trace --detailed sw0 '...'
# ── 出力例 ──
ingress(dp="sw0", inport="sw0-port1")
------------------------------------
 0. ls_in_port_sec_l2 (northd.c:XXXX): inport == "sw0-port1" && eth.src == {00:00:00:00:00:01}, priority 50
    next;
 5. ls_in_pre_acl (northd.c:XXXX): ip, priority 100
    reg0[0] = 1;
    next;
 7. ls_in_acl_hint (northd.c:XXXX): !ct.new && ct.est && !ct.rpl && ct_mark.blocked == 0, priority 4
    reg0[8] = 1; reg0[10] = 1; next;
 9. ls_in_acl (northd.c:XXXX): reg0[10] == 1 && (outport == "sw0-port2" && ip4 && tcp && tcp.dst == 80), priority 2002
    next;                                   ← ACL 通過（allow-related）
...
egress(dp="sw0", inport="sw0-port1", outport="sw0-port2")
---------------------------------------------------------
 9. ls_out_port_sec_l2: outport == "sw0-port2" && eth.dst == {00:00:00:00:00:02}, priority 50
    output;                                 ← ここに出て行く
    /* output to "sw0-port2", type "" */
  • ingress(dp=...) / egress(dp=...): パイプラインの境目。ここが切り替わらずに終わる = どこかでドロップ
  • テーブル番号（0〜32） と ステージ名（ls_in_pre_acl など）: OVN のパイプライン段階。名前だけで「ACL 判定中」「NAT 適用中」が分かる
  • match 式 + priority: 実際にヒットした条件とその優先度
  • action: next;（次段へ）/ output;（出力）/ drop;（破棄）/ ct_next;（conntrack 呼び出し）

実例 1: 「ACL でブロックされているかを確認」

# VM-A (sw0-port1, 10.0.0.2) から VM-B (sw0-port2, 10.0.0.3) の TCP/22 を試す
sudo ovn-trace --summary sw0 \
  'inport == "sw0-port1" && eth.src == 00:00:00:00:00:01 \
   && eth.dst == 00:00:00:00:00:02 && ip4 && ip4.src == 10.0.0.2 \
   && ip4.dst == 10.0.0.3 && ip.ttl == 64 && tcp && tcp.dst == 22'

── ドロップしている場合の典型的な出力 ──
ingress(dp="sw0", inport="sw0-port1")
    next(pipeline=ingress, table=8);    // ls_in_acl
    ls_in_acl: ip4 && tcp && tcp.dst == 22, priority 2001, drop;
*** drop action

drop の手前にヒットした ACL ルールの priority が見えるので、NB DB でその ACL を特定できます:

ovn-nbctl acl-list sw0
from-lport 2001 (ip4 && tcp && tcp.dst == 22) drop  ← これが原因
from-lport 1000 (ip4) allow-related

実例 2: 論理ルータと NAT の挙動確認

# 内部 VM (10.0.0.2) から外部 (8.8.8.8:443) へ出る想定。SNAT 定義を検証
sudo ovn-trace --detailed ls-internal \
  'inport == "vm1-port" && eth.src == 00:11:22:33:44:55 \
   && eth.dst == 00:aa:bb:cc:dd:ee && ip4 \
   && ip4.src == 10.0.0.2 && ip4.dst == 8.8.8.8 \
   && ip.ttl == 64 && tcp && tcp.dst == 443'

── 重要部分 ──
lr_in_ip_routing: reg7 == 0 && ip4.dst == 0.0.0.0/0, priority 1, reg0 = 192.0.2.1; ...
lr_in_arp_resolve: ... next;
lr_out_snat: ip4.src == 10.0.0.0/24, priority 50,
  ct_snat(192.0.2.100);        ← SNAT 後の IP がこれで合っているか確認
output;

実例 3: Load Balancer（VIP → backend）の展開を見る

sudo ovn-trace --detailed lr-public \
  '... && ip4.dst == 192.0.2.50 && tcp && tcp.dst == 80'

── LB の動き ──
lr_in_dnat: ip4.dst == 192.0.2.50 && tcp.dst == 80, priority 120,
  ct_lb_mark(backends=10.0.0.10:8080,10.0.0.11:8080);
← VIP (192.0.2.50:80) を backend どちらかにハッシュで振り分けていることを確認

ovn-trace / ovs-appctl ofproto/trace / conntrack の使い分け

やりたいこと 使うコマンド
「このパケットを流したら、論理的にどこで落ちる？」 ovn-trace（NB DB 設定の論理検証）
「OVS の OpenFlow パイプラインとしてどう処理される？」 ovs-appctl ofproto/trace br-int "..."
「実際に流してみて、どのフローにカウンタが乗った？」 ovs-ofctl dump-flows 前後比較
「今、kernel に残っている接続は？」 conntrack -L

3 つを組み合わせた実戦デバッグ

状況: OpenStack + OVN 環境で、VM-A (10.0.0.2) から VM-B (10.0.0.5, TCP/443) への通信が 50% くらいの確率で RST になる、という曖昧な障害。以下の手順で切り分けます。

論理的に通るはず？ を ovn-trace で確認

sudo ovn-trace --summary ls-tenant1 'inport == "vm-a" && ... && tcp.dst == 443'

→ output で VM-B のポートに出る判定なら、論理設定は OK。ACL / ルーティングの問題はここで除外できる。

実フローにカウンタが乗っているかを ovs-ofctl dump-flows で

# ping の前後で差分を取る
sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | grep nw_dst=10.0.0.5 > /tmp/before
# （ここで VM-A → VM-B に 100 回 curl など）
sudo ovs-ofctl -O OpenFlow13 dump-flows br-int | grep nw_dst=10.0.0.5 > /tmp/after
diff /tmp/before /tmp/after

→ allow フローの n_packets は増えているが、別テーブルで drop フローも増えていた場合 = priority 衝突。
→ 増えているフローの cookie から原因の Logical_Flow を特定できる。

kernel 側で接続が残っているかを conntrack -L で

sudo conntrack -L -p tcp --dport 443 -s 10.0.0.2 -d 10.0.0.5

→ ESTABLISHED が見えるなら commit は効いているが、RST が混ざるなら conntrack 側で INVALID 判定を食っている可能性（例: TCP Out-of-Window）。 sysctl net.netfilter.nf_conntrack_tcp_be_liberal=1 で緩和できる場合あり。

結論と対処

「論理は OK、フロー配布も OK、ただし kernel 側で TCP 状態が厳しくドロップ判定になっていた」という切り分けが、3 コマンドを順に使って 10 分で完結します。どれか 1 つでは「論理は正しい / 実際に動いている / kernel は許可している」のどの層で詰まっているかが見えません。

鉄則: 上から降りる（ovn-trace → ovs-ofctl dump-flows → conntrack -L）。論理の誤りは物理で直せないし、物理で動いていないものは kernel には届きません。

まとめ

  • conntrack -L: Linux kernel の接続追跡テーブルを実測。NAT 変換結果・接続 state・table full の原因を把握する。
  • ovs-ofctl dump-flows: OVS の OpenFlow テーブルを実測。n_packets が「そのフローにパケットが到達したか」を語ってくれる。
  • ovn-trace: OVN の論理パイプラインをシミュレート。パケットを流さずに机上検証でき、論理設計の正しさを確認できる。
  • 3 つは同じネットワーク障害を別々のレイヤから見る補完関係にある。上（論理） → 中（物理フロー） → 下（kernel 接続）の順に降ろすのが定石。
  • どのコマンドも単独の出力を「眺める」だけでは情報不足で、フィルタ（状態・プロトコル・IP・ゾーン）と差分観察を組み合わせて初めて真価を発揮する。
