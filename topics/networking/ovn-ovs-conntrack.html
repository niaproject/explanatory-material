OVN と OVS の違い、そして conntrack の役割

Open vSwitch / Open Virtual Network / Linux conntrack の関係を整理

このページの結論（TL;DR）

3 行でまとめると

OVS（Open vSwitch）は 1 ホスト上で動くデータプレーンの仮想スイッチ。OVN（Open Virtual Network）はその上位に乗るクラスタ全体のコントロールプレーンで、論理スイッチ / ルータ / ACL / LB / NAT を宣言的に扱うレイヤです。OVN は単体では動かず、必ず OVS を前提とします。

conntrack は Linux カーネルの nf_conntrack が提供する接続追跡機能。OVS の kernel datapath が OpenFlow の ct() アクション経由で直接呼び出し、OVN はそれを「ステートフル ACL」「Load Balancer」「NAT」といった論理機能として抽象化します。

つまり conntrack は OVN ではなく OVS（と Linux kernel）のレイヤで実行され、OVN は「その結果をどう使いたいか」を宣言するだけ、という役割分担になっています。

OVS (Open vSwitch) とは

Open vSwitch (OVS) は 2007〜2008 年に Nicira Networks でスタートしたオープンソースの仮想スイッチです。 2012 年に VMware が Nicira を買収し、現在は Linux Foundation の下で開発が継続されています。 カーネルモジュールが GPL v2、メインコードは Apache 2.0、一部に BSD が含まれるライセンス構成で、 現行安定版は OVS 3.7.0（2026 年 2 月リリース）です。

アーキテクチャ構成

OVS は「ユーザー空間デーモン + カーネルモジュール」の二層構造で動作します。

OVS の内部コンポーネント
ovs-vswitchd ユーザー空間 / メインデーモン / OpenFlow 処理
ovsdb-server ユーザー空間 / JSON-RPC DB サーバー
▼ upcall / フロー登録
kernel datapath openvswitch.ko / フローキャッシュ / 高速転送
userspace datapath DPDK / AF_XDP / ポーリングで高速化

パケット処理の流れ

  1. パケット到着 → kernel datapath がフローキャッシュを検索
  2. キャッシュヒット → そのまま高速転送（ユーザー空間に戻らない）
  3. キャッシュミス → upcall でユーザー空間の ovs-vswitchd に問い合わせ → OpenFlow パイプライン実行 → 結果をキャッシュに登録 → 以降の同一フローは kernel のみで処理

L2 / トンネル

VLAN / VXLAN / Geneve / GRE / STT をサポート。

OpenFlow

OpenFlow 1.0〜1.5 に対応。外部コントローラから制御可能。

可観測性

NetFlow / sFlow / IPFIX によるフロー統計。

QoS

HTB / HFSC などのキューイング。

conntrack 統合

OpenFlow の ct() アクションで Linux nf_conntrack を直接呼び出し。

主なユースケース

  • KVM / libvirt の仮想マシンネットワーク
  • OpenStack Neutron（ML2/OVS ドライバ）
  • Kubernetes CNI プラグイン（Antrea など）
  • SDN コントローラー（ONOS / OpenDaylight）との連携

OVN (Open Virtual Network) とは

Open Virtual Network (OVN) は 2015 年 1 月に OVS コアチーム （Justin Pettit、Ben Pfaff ら）が発表した、OVS の上位に乗る コントロールプレーンです。OVN は OVS 抜きでは動作しません。

OVN が解決した課題

  • マルチホストにまたがる論理ネットワークの自動構成（OVS 単体では各ホスト個別設定が必要）
  • L3 ルーティングを分散実装（従来は集中ルータ VM がボトルネック）
  • セキュリティグループの効率的な実装（OpenFlow 手書きは量とバグが問題）
  • ロードバランサ / DHCP の分散実装

アーキテクチャ

OVN の NB → SB → OVS データフロー
CMS（OpenStack / Kubernetes など） 論理設定を書き込む
▼
OVN NB DB Logical_Switch / Logical_Router / ACL / Load_Balancer / NAT（人間が扱う論理設定）
▼ ovn-northd が翻訳
ovn-northd 中央集権プロセス / NB → SB 変換エンジン / Logical Flow 生成
▼
OVN SB DB Logical_Flow（match/action 中間表現）+ Port_Binding + Chassis
▼ 各ホストの ovn-controller が読む
ovn-controller（各ハイパーバイザ） SB の Logical Flow を OpenFlow に変換して OVS に投入
▼
OVS（ovs-vswitchd + kernel datapath） 実パケットを転送
Logical Flow とは: OVN 独自の match/action 中間表現で、 物理ポート情報や個別ホストの事情を含まないため、 クラスタ全体で共通化できます。ovn-controller が各ホストの物理情報 （ローカルポート番号、トンネル先 IP 等）を埋めて OpenFlow に落とし込みます。

分散ゲートウェイ

各ホストでローカル応答。中央サーバ不要。

conntrack とは

conntrack (Connection Tracking) は Linux カーネルの netfilter サブシステムの一部で、2.6.15 以降に nf_conntrack として実装されています。 パケットが通るたびに、それがどの「接続」に属するかを5 タプル単位で追跡し、 ステートフルなファイアウォール / NAT / ロードバランサの基盤を提供します。

5 タプル

src_ip : dst_ip : sport : dport : proto

対応プロトコル: TCP / UDP / ICMP / DCCP / SCTP / GRE。

主な状態

NEW ESTABLISHED RELATED INVALID UNTRACKED
状態意味
NEW最初のパケットを見た状態（まだ応答未観測）
ESTABLISHED往復両方向のパケットを観測済み。正常な接続
RELATED既存接続に関連するパケット（FTP データコネクション、ICMP エラー等）
INVALID既存接続にも属さず新規でもない。通常は破棄対象
UNTRACKEDNOTRACK ターゲット等で明示的に追跡除外されたパケット

主要コマンド（conntrack-tools）

# 全エントリの表示
conntrack -L

# 状態で絞り込み（ESTABLISHED の TCP のみ）
conntrack -L -p tcp --state ESTABLISHED

# エントリ削除（5 タプル指定）
conntrack -D -s 10.0.0.1 -d 10.0.0.2 -p tcp --sport 5000 --dport 80

# 現在のエントリ数と設定値の確認
cat /proc/sys/net/netfilter/nf_conntrack_count
cat /proc/sys/net/netfilter/nf_conntrack_max

# 旧方式のテーブル生表示（procfs）
cat /proc/net/nf_conntrack

主な sysctl チューニングパラメータ

パラメータ意味既定値 / 目安
nf_conntrack_max 追跡できる最大エントリ数 総メモリ / 16384（最小 65536）／ 8 GB RAM なら 262144 程度が目安
nf_conntrack_buckets 内部ハッシュテーブルのバケット数 通常 max / 4 推奨
nf_conntrack_tcp_timeout_established ESTABLISHED 状態のエントリ保持時間 既定 432000 秒（= 5 日）。LB 前段ホストでは短縮検討

代表的なトラブル

nf_conntrack: table full, dropping packet

エントリが nf_conntrack_max に達した状態。新規接続がドロップされ、 既存の追跡済み接続は維持されます（= 新規 SYN だけが落ちる症状）。 K8s ノードや LB でよく見る。

対処の順序

  1. nf_conntrack_max を増やす（ただし RAM を食うのでほどほどに）
  2. TCP timeout_established を短縮する（長寿命のゴミエントリを減らす）
  3. 大量 UDP などは NOTRACK / ct_state=-trk で追跡除外
  4. Prometheus 等で nf_conntrack_count を監視してアラート

conntrack は OVN と OVS のどちらと関係するのか

結論: conntrack は OVS（と Linux kernel）の責務。 OpenFlow の ct() アクション経由で kernel の nf_conntrack を直接呼び出します。 OVN は conntrack を直接意識せず、「ステートフル ACL」「LB」「NAT」として論理的に宣言するだけで、裏側で ovn-northd が ct_commit 等を含む Logical Flow を生成し、最終的に OVS の ct() に落ちる形です。

OVS に入った ct() アクション

2016 年頃の OVS に ct() アクションが実装され、それまで必要だった 「Linux bridge + iptables」に一度迂回してから OVS に戻すという煩雑なデータパスが不要になりました。 OVS 自身が OpenFlow パイプラインの中から直接 nf_conntrack を呼べます。

ct() アクションの主なパラメータ

ct(table=N, commit, zone=N, exec(set_field:N->ct_mark), nat)
パラメータ意味
table=Nconntrack 処理後のパケットを table N に再投入
commitこの接続を conntrack テーブルに永続化（以降 ESTABLISHED 化）
zone=N16 bit のテナント識別子。重複 IP 空間を持つマルチテナント対応
exec(...)conntrack エントリにメタデータ（ct_mark, ct_label）を付与
natSNAT / DNAT を conntrack と連携させて実行

ステートフルファイアウォール実装例（OpenFlow）

「左ポート（veth_l0）から右ポート（veth_r0）への接続は許可、戻りも許可、それ以外の戻りは遮断」を conntrack で実装した例。

未追跡パケットを conntrack に送る

# ct_state=-trk: まだ conntrack を通っていないパケット
table=0, priority=50, ct_state=-trk, tcp, in_port=veth_l0,
  actions=ct(table=0)

NEW 接続は commit して転送

# ct_state=+trk+new: 追跡済みかつ新規
table=0, priority=50, ct_state=+trk+new, tcp, in_port=veth_l0,
  actions=ct(commit), output:veth_r0

ESTABLISHED の応答は自動で許可

# ct_state=+trk+est: 確立済み接続の戻りパケット
table=0, priority=50, ct_state=+trk+est, tcp, in_port=veth_r0,
  actions=output:veth_l0
ポイント: 戻りトラフィックに対して個別の許可ルールを書く必要がない （= セキュリティグループのルール数が激減）のが conntrack の最大のメリットです。

OVN は ct() を直接書かない

OVN では ovn-nbctl acl-add ... allow-related のように論理的な宣言を書くだけで、 ovn-northd が自動的に ct_commit を含む Logical Flow を生成し、 各ホストの ovn-controller がそれを OpenFlow の ct() に変換します。

ステートフル ACL

OVN のステージ ls_in_stateful / lr_in_stateful で conntrack 処理が挟まれる。allow-related で戻りを自動許可。

Load Balancer

VIP:Port を Load_Balancer テーブルに登録するだけ。 conntrack DNAT で各バックエンドに分散、各ホストで分散実行。

NAT

NAT テーブル定義のみ。集中型 GW ルータ、または分散 GW ポートで実行。 戻りの逆変換は conntrack 任せ。

conntrack を使うメリット（OVS/OVN 共通）

  1. 接続状態管理（TCP ステートマシン含む）をカーネル任せにでき、自前実装が不要
  2. カーネルのハッシュテーブル最適化で O(1) ルックアップ
  3. NAT の逆変換が自動（戻りに対する OpenFlow ルールが不要）
  4. セキュリティグループのルール数が激減（戻りは ESTABLISHED で自動許可）
  5. zone= によるマルチテナント重複 IP 対応

レイヤ対応図（NB DB の 1 行がどこまで降りていくか）

OVN NB DB:ACL allow-related
└▶ovn-northd:Logical Flow に ct_commit を生成
　└▶OVN SB DB:Logical_Flow 行として保存
　　└▶ovn-controller:OpenFlow ct(commit, table=N, zone=M)
　　　└▶ovs-vswitchd:OpenFlow パイプライン実行
　　　　└▶kernel datapath:nf_conntrack 呼び出し
　　　　　└▶Linux kernel:nf_conntrack テーブルにエントリ記録
覚えておくと良いこと: conntrack -L で見えるエントリはあくまで Linux kernel の持ち物。 OVN / OVS のプロセスが落ちても、カーネルが動いている限り conntrack 自体は生きている、というレイヤ構造です。

まとめ

  • OVS = データプレーン（1 ホスト上の仮想スイッチ）、OVN = コントロールプレーン（クラスタ全体の論理ネット管理）。OVN は OVS 無しでは動かない。
  • OVN は ovn-nbctl で論理的に書き、ovn-northd が Logical Flow、ovn-controller が各ホストで OpenFlow に落とす。
  • conntrack は Linux kernel の nf_conntrack。5 タプルで接続を追跡し、ステートフル FW / NAT / LB の基盤になる。
  • OVS は OpenFlow の ct() アクションでカーネル conntrack を直接呼び出す。OVN は ct() を書かず、ACL / LB / NAT として宣言するだけ。
  • トラブル対応では conntrack -L / ovs-ofctl dump-flows / ovn-trace をレイヤごとに切り分けて使うのが基本。
