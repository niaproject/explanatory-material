自動モード vs バイパス権限モード

Claude Code の権限モードの違いを、許可プロンプトの挙動と安全性の観点から整理

結論（3 行まとめ）

  • 自動モード（acceptEdits） = ファイル編集だけ自動承認。Bash などコマンド実行は毎回ユーザーに確認する。
  • バイパス権限モード（bypassPermissions） = すべての許可チェックをスキップ。ファイル編集もシェルコマンドも確認なしで実行される。
  • 編集のレビュー負担を減らしたいなら 自動モード、本当に安全な隔離環境で全自動にしたいときだけ バイパス、が基本方針。

そもそも Claude Code の「権限モード」とは

Claude Code はツール実行（ファイル書き込み、シェルコマンド実行、ウェブフェッチなど）ごとに、 設定で定義された 許可ルール を評価して「自動許可」「ユーザーに確認」「拒否」を決めます。 権限モードは、この評価ロジック全体に被さるスイッチで、個別の許可ルールを上書きしたり緩めたりします。

モードは Shift+Tab で default → acceptEdits → plan → default → … のようにサイクル切り替えできます。 plan モード（書き込み一切禁止の読み取り専用）もあり、読む専門・設計フェーズ向けです。

補足: bypassPermissions は Shift+Tab の通常サイクルには入りません。 明示的に起動オプションや設定で指定しない限り有効にならない設計になっています（後述）。

起動方法と切り替え

自動モード（acceptEdits）

最も手軽。Claude Code を普通に起動したあと、セッション中に切り替えます。

normal → Shift+Tab → ⏵ auto-accept edits on

入力欄の下（ステータスバー）に ⏵⏵ auto-accept edits on のような緑色の表示が出ればオンです。 もう一度 Shift+Tab を押すと plan モードに、さらに押すと default に戻ります。

起動時から自動にしたい場合は --permission-mode で指定できます。

# 起動時から acceptEdits にする
claude --permission-mode acceptEdits

バイパス権限モード（bypassPermissions）

Shift+Tab のサイクルには含まれません。うっかり入らないように、 CLI フラグまたは設定ファイルで明示的に選ぶ必要があります。 

# 起動時にバイパスモードを有効化（警告が出る）
claude --dangerously-skip-permissions

# もしくは --permission-mode での指定（同義）
claude --permission-mode bypassPermissions
⚠ bypass permissions 全ツール確認スキップ中
⚠ 実行前の確認: --dangerously-skip-permissions は、 起動時に 「Docker / VM / サンドボックス環境でのみ使うことを推奨する」 という趣旨の警告が出ます。 ホスト OS でそのまま流すと、意図せず
rm -rf ~/ / git push --force origin main / aws s3 rm --recursive のような破壊的コマンドが確認なしで走る可能性があります。

シナリオで見る挙動の違い

同じ指示「この関数をリファクタして、単体テストを実行して」を 3 つのモードで流したときの流れ。

default モード

User 「関数 foo をリファクタして、テストして」 default 📝
Claude Edit foo.ts の実行許可を求める 確認 ⏸
User 差分を見て Approve 確認 ✅
Claude Bash: npm test の実行許可を求める 確認 ⏸
User コマンド内容を見て Approve 確認 ✅

acceptEdits モード（自動）

User 「関数 foo をリファクタして、テストして」 auto 📝
Claude Edit foo.ts を即時適用（差分表示のみ） 自動 ⚡
Claude Bash: npm test の実行許可を求める 確認 ⏸
User コマンド内容を見て Approve 確認 ✅

bypassPermissions モード（バイパス）

User 「関数 foo をリファクタして、テストして」 bypass 📝
Claude Edit foo.ts を即時適用 無確認 ⚡
Claude Bash: npm test をそのまま実行 無確認 ⚡
Claude テスト失敗 →「一度全部消してやり直します」と rm -rf src/ を実行（!） 無確認 💥
これが怖いところ: バイパスモードでは Claude が「作業効率のため」と判断した破壊的な操作も、 ユーザーに確認されません。モデルが誤判断した瞬間、ファイルが消えたり Git 履歴が吹き飛んだりします。 ロールバック可能な環境（Git コミット済み・サンドボックス・コンテナ）でのみ使うのが鉄則。

いつどちらを使うか

🟢 自動モードがハマる場面
  • 定型的なリファクタ・リネームなど、多数のファイルに軽微な編集を加える作業
  • ドキュメント・コメント修正など副作用ゼロの編集
  • CI/CD やテスト実行は手動承認したいが、編集の承認は面倒
  • 差分をコミット前にレビューする前提で流し込む
🟢 バイパスモードが（限定的に）有効な場面
  • Docker / devcontainer の中など、ホストから完全隔離された環境での一括作業
  • 使い捨てのサンドボックス VM で検証用コードを生成する
  • CI 上で Claude Agent SDK を動かすバッチ処理（入出力が明確に制限されている前提）
  • Git で状態を常に巻き戻せるブランチ上でのみの作業
🔴 バイパスモードを使ってはいけない場面
  • ホスト OS の開発環境（自分の PC の日常作業）
  • 本番のクラウド認証情報（AWS / GCP / Azure）が有効なシェル
  • SSH 鍵・API トークン・機密情報を扱うリポジトリ
  • コミットされていない未バックアップの変更がある状態
  • 本番 DB や共有インフラに接続可能な環境
🔴 自動モードでも油断しない
  • 編集対象が設定ファイル・CI 定義だと副作用が大きい
  • package.json / pyproject.toml などの依存変更は確認すべき
  • 大量の編集が一気に流れるので、コミット粒度を細かく保つ

安全に使うためのチェックリスト

  • 作業開始前に git status がクリーン、または直前のコミットが push/stash 済みであること。
  • バイパスモードはコンテナ/VM の中だけ。ホスト OS で --dangerously-skip-permissions を常用しない。
  • 自動モードでも、差分を途中で一度は目視する（Esc で止めて確認）。
  • 機密情報を含むリポジトリには .claude/settings.json で deny ルールを明示する。
  • バイパスモードを使うセッションでは クラウド認証情報を別プロファイルに隔離する。
  • 「全自動化したい」動機なら、まず Claude Agent SDK や CI 上の実行で代替できないかを検討する。

参考: 設定ファイルからの指定

プロジェクト固有の .claude/settings.json または ユーザー設定 ~/.claude/settings.json で、 デフォルトの権限モードを指定できます。 

// ~/.claude/settings.json — 起動時に常に acceptEdits にする
{
  "permissions": {
    "defaultMode": "acceptEdits"
  }
}
bypassPermissions の設定: 同様に "defaultMode": "bypassPermissions" を書けば常に有効化できますが、 推奨されません。特定の devcontainer / プロジェクト限定にしたいなら、プロジェクトルートの .claude/settings.json に書くのが安全です （ユーザー設定に書くと全プロジェクトで有効になる）。
許可ルールで賢く絞る: そもそも「特定のコマンドだけ自動許可」で十分なら、 permissions.allow / permissions.deny に個別ルールを書く方が安全です。 例えば "Bash(npm test)" を allow に入れれば、テスト実行だけは確認なしで流れ、 それ以外の Bash は default 通り確認されます。バイパスモードは最後の手段。

「編集のレビュー疲れを解消したい」なら 自動モード、 「完全に隔離された環境で全自動で動かしたい」ならだけ バイパス、 という切り分けを徹底すれば大事故は避けられます。
